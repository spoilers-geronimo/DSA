Formal Definition

Dynamic Programming (DP) is an algorithmic paradigm that solves complex optimisation and counting problems by breaking them into overlapping subproblems, solving each subproblem exactly once, and storing the result for future reuse. It applies when a problem has two properties: overlapping subproblems — the same subproblem recurs multiple times in the naive recursive solution — and optimal substructure — the optimal solution to the whole problem can be constructed from optimal solutions to its subproblems. DP can be implemented top-down via memoised recursion or bottom-up via iterative tabulation.

DP is often described as "recursion with a cache." The cache (a hash map or array) ensures that each subproblem is solved only once — transforming exponential brute-force recursion into polynomial time. The name "Dynamic Programming" was coined by Richard Bellman in 1953; it is deliberately vague, chosen to obscure the mathematical nature of the work from government funding committees.

Plain English Explanation & Patterns

The core idea: don't solve the same problem twice. If a recursive function is called with the same arguments repeatedly, you're wasting work. Store the answer the first time and look it up instantly every time after. That's the entirety of memoisation.

DP has two implementation styles and two common problem structures:

⬇️ Top-Down (Memoisation)

  • Write the natural recursive solution first
  • Add a cache (Map or object) to store results
  • Before computing, check if result is already cached
  • Easy to write — hard to reason about space
  • Solves only the subproblems actually needed

⬆️ Bottom-Up (Tabulation)

  • Build an array (the DP table) from base cases upward
  • Fill each cell using previously filled cells
  • No recursion, no call stack overhead
  • Easier to optimise space (rolling array)
  • Solves all subproblems, even unused ones

📐 1D DP

  • State is a single integer: dp[i]
  • Each value depends on a few previous values
  • Examples: Fibonacci, Climbing Stairs, House Robber
  • Often optimised to O(1) space with two variables

🔲 2D DP

  • State is two integers: dp[i][j]
  • Common when two sequences or two constraints exist
  • Examples: Longest Common Subsequence, Knapsack, Edit Distance
  • Often optimised to O(n) space with a rolling row

The DP framework — four questions to ask any DP problem:

  • 1
    What is the state? What variables fully describe a subproblem? Usually indices, counts, or capacities. e.g., dp[i] = best answer for the first i items.
  • 2
    What is the recurrence? How do I compute dp[i] from earlier values? This is the core transition. e.g., dp[i] = max(dp[i-1], dp[i-2] + val[i]).
  • 3
    What are the base cases? The smallest subproblems I can answer directly, without using the recurrence. e.g., dp[0] = 0, dp[1] = val[0].
  • 4
    What is the answer? Which cell in the DP table holds the final answer? Usually dp[n] or dp[n][W].
Classic Problem DP Type Time Space
Fibonacci / Climbing Stairs 1D linear O(n) O(1)
House Robber 1D linear, skip constraint O(n) O(1)
Coin Change 1D knapsack-style O(n × amount) O(amount)
Longest Common Subsequence 2D sequence O(m × n) O(min(m,n))
0/1 Knapsack 2D capacity O(n × W) O(W)
Edit Distance 2D sequence O(m × n) O(min(m,n))
💡

DP vs Greedy vs Divide-and-Conquer: All three break problems into smaller parts. Greedy makes one irrevocable local choice at each step — fast but only correct for certain problem structures. Divide-and-conquer splits into independent subproblems with no overlap — merge sort is the classic example. DP handles overlapping subproblems and builds optimal solutions from all possibilities — more expensive but universally applicable when the two DP properties hold.

⚠️

The most common DP mistake: writing a memoised solution but using a mutable object (like an array) as a cache key. Objects in JavaScript are compared by reference, not value — [1,2] !== [1,2]. Always use primitive keys (numbers, strings) for your memo map. For 2D problems, use `${i},${j}` as the key or a 2D array.

Code Snippet

Two complete DP solutions — one 1D, one 2D — each shown in both top-down and bottom-up style so you can see how they relate.

JavaScript — Coin Change (1D DP) 
// ─── Coin Change — LeetCode #322 ────────────────────────────────
// Given coins of different denominations and a target amount,
// find the minimum number of coins to make up the amount.
// coins = [1, 2, 5], amount = 11  →  3 coins (5+5+1)

// State: dp[i] = minimum coins needed to make amount i
// Recurrence: dp[i] = min over all coins c of (dp[i - c] + 1)
// Base case: dp[0] = 0  (0 coins needed for amount 0)

// ─── Bottom-Up Tabulation — O(amount × coins) ───────────────────
function coinChange(coins, amount) {
  // Fill table with Infinity (impossible to reach)
  const dp = new Array(amount + 1).fill(Infinity);
  dp[0] = 0;   // base case

  for (let i = 1; i <= amount; i++) {
    for (const coin of coins) {
      if (i - coin >= 0) {
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }
  return dp[amount] === Infinity ? -1 : dp[amount];
}

console.log(coinChange([1, 2, 5], 11));   // 3  (5+5+1)
console.log(coinChange([2],         3));    // -1 (impossible)

// ─── dp table for coins=[1,2,5], amount=7 ────────────────────────
// i:    0   1   2   3   4   5   6   7
// dp:   0   1   1   2   2   1   2   2
//           ↑   ↑               ↑   ↑
// dp[1]=dp[0]+1   dp[5]=dp[0]+1   dp[7]=dp[5]+dp[2]=1+1=2
JavaScript — Longest Common Subsequence (2D DP) 
// ─── Longest Common Subsequence (LCS) — LeetCode #1143 ──────────
// Given two strings, return the length of their LCS.
// "abcde" and "ace" → 3  (the subsequence "ace")

// State: dp[i][j] = LCS length of text1[0..i-1] and text2[0..j-1]
// Recurrence:
//   if text1[i-1] === text2[j-1]:  dp[i][j] = dp[i-1][j-1] + 1
//   else:                           dp[i][j] = max(dp[i-1][j], dp[i][j-1])
// Base case: dp[i][0] = dp[0][j] = 0 (empty string → LCS length 0)

function longestCommonSubsequence(text1, text2) {
  const m = text1.length, n = text2.length;
  // Create (m+1) × (n+1) table, filled with 0
  const dp = Array.from({ length: m + 1 }, () => new Array(n + 1).fill(0));

  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (text1[i - 1] === text2[j - 1]) {
        dp[i][j] = dp[i - 1][j - 1] + 1;   // characters match: extend LCS
      } else {
        dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);  // take best of skip
      }
    }
  }
  return dp[m][n];
}

console.log(longestCommonSubsequence("abcde", "ace"));   // 3
console.log(longestCommonSubsequence("abc",   "abc"));   // 3
console.log(longestCommonSubsequence("abc",   "def"));   // 0

Plain English Analogy

📝

A Student Who Never Recalculates the Same Maths Problem Twice

Imagine a maths student solving a complex problem in an exam. Partway through, she realises that computing the answer requires knowing the value of x² for several different values of x. A careless student would compute 7² = 49 every time it appears in the problem — maybe five times over. A smart student computes 7² = 49 once, writes it in the margin, and looks it up the next four times. Same answer. A fraction of the work.

That is memoisation. The margin notes are the cache (the DP table). The first time you solve a subproblem, you write the answer down. Every subsequent time you need it, you read it from the margin instead of recomputing it.

Bottom-up DP is the same student but even more organised. Before starting the big problem, she pre-computes and writes down every square from 1² to 20² in a table at the top of her scratch paper. When she needs any value, it's always there. She never has to compute anything twice.

  • The exam problem = the main problem to solve
  • Intermediate calculations (x²) = overlapping subproblems
  • Writing in the margin = memoisation (top-down)
  • Pre-filling a table = tabulation (bottom-up)
  • "Have I seen this before?" = the cache lookup
  • The final answer = dp[n] or dp[n][W]

Real-World Coding Applications

Dynamic programming is the backbone of optimisation across every domain of computing — anywhere the best solution to a large problem depends on best solutions to smaller ones.

🧬

DNA Sequence Alignment

Bioinformatics tools like BLAST and ClustalW align DNA/protein sequences using DP (the Smith-Waterman and Needleman-Wunsch algorithms). Edit Distance DP finds the minimum number of mutations between two sequences.

🗣️

Auto-correct & Spell-Check

Autocorrect computes the Edit Distance (Levenshtein distance) between your typed word and every word in the dictionary. DP makes this fast enough to run in real time on a phone keyboard on every keystroke.

📦

Shipping & Logistics

The Knapsack problem models packing a cargo plane or delivery truck: given items with weights and values, maximise value under a weight constraint. FedEx, UPS, and Amazon's routing optimisation all use DP variants.

📈

Stock Trading Algorithms

The "Best Time to Buy and Sell Stock" family of problems (LeetCode #121–#188) are all DP. Financial algorithms for optimal order execution, portfolio rebalancing, and options pricing (Black-Scholes) use DP on discrete state grids.

🌐

Network Routing (Bellman-Ford)

The Bellman-Ford shortest-path algorithm is a classic DP over the graph's edges. It finds shortest paths in networks with negative edge weights and is used in internet routing protocols like RIP (Routing Information Protocol).

🤖

Reinforcement Learning

Value iteration and policy iteration — the foundational algorithms for training reinforcement learning agents — are DP algorithms on the Markov Decision Process state space. AlphaGo's training pipeline is DP at its deepest level.

Classic Interview Question

🏠 House Robber

You are a robber planning to rob houses along a street. Each house has a certain amount of money. You cannot rob two adjacent houses (the alarm would trigger). Given an integer array nums representing the money in each house, return the maximum amount you can rob tonight.

Example: nums = [2, 7, 9, 3, 1] → 12 (rob houses 0, 2, 4: 2+9+1=12)

🧩 Medium — LeetCode #198 · the cleanest 1D DP problem in existence

Why this question? House Robber is the perfect introduction to 1D DP. The constraint — can't take adjacent houses — forces you to define a state and a recurrence. Once you see it, the same pattern unlocks House Robber II (circular street), Minimum Cost Climbing Stairs, Maximum Alternating Subsequence, and dozens more.

The key insight: At each house i, you have two choices — rob it (and take whatever you robbed up to house i-2) or skip it (and keep whatever you got up to house i-1). Pick the bigger of the two.

  • 1
    State: dp[i] = maximum money robbed from the first i houses.
  • 2
    Recurrence: dp[i] = max(dp[i-1], dp[i-2] + nums[i]) — either skip house i or rob it (adding to the best result two houses back).
  • 3
    Base cases: dp[0] = nums[0], dp[1] = max(nums[0], nums[1]).
  • 4
    Space optimisation: Only dp[i-1] and dp[i-2] are ever needed — replace the array with two variables for O(1) space.
JavaScript — House Robber: Four Approaches 
// ─── Approach 1: Naïve Recursion — O(2ⁿ) ────────────────────────
function robNaive(nums, i = nums.length - 1) {
  if (i < 0)  return 0;
  if (i === 0) return nums[0];
  return Math.max(
    robNaive(nums, i - 1),              // skip house i
    robNaive(nums, i - 2) + nums[i]    // rob house i
  );
}

// ─── Approach 2: Memoised — O(n) time, O(n) space ────────────────
function robMemo(nums, i = nums.length - 1, memo = {}) {
  if (i < 0)   return 0;
  if (i === 0)  return nums[0];
  if (memo[i] !== undefined) return memo[i];
  memo[i] = Math.max(
    robMemo(nums, i - 1, memo),
    robMemo(nums, i - 2, memo) + nums[i]
  );
  return memo[i];
}

// ─── Approach 3: Bottom-Up DP — O(n) time, O(n) space ────────────
function robDP(nums) {
  if (nums.length === 1) return nums[0];
  const dp = new Array(nums.length);
  dp[0] = nums[0];
  dp[1] = Math.max(nums[0], nums[1]);
  for (let i = 2; i < nums.length; i++) {
    dp[i] = Math.max(dp[i - 1], dp[i - 2] + nums[i]);
  }
  return dp[nums.length - 1];
}

// ─── Approach 4: Space-Optimised — O(n) time, O(1) space ─────────
function rob(nums) {
  let prev2 = 0, prev1 = 0;
  for (const n of nums) {
    const curr = Math.max(prev1, prev2 + n);
    prev2 = prev1;
    prev1 = curr;
  }
  return prev1;
}

// ─── Test all four ────────────────────────────────────────────────
const nums = [2, 7, 9, 3, 1];
console.log(robNaive(nums));  // 12
console.log(robMemo(nums));   // 12
console.log(robDP(nums));     // 12
console.log(rob(nums));       // 12

// dp trace for [2,7,9,3,1]:
// i=0: dp[0]=2
// i=1: dp[1]=max(2,7)=7
// i=2: dp[2]=max(7, 2+9)=11
// i=3: dp[3]=max(11,7+3)=11
// i=4: dp[4]=max(11,11+1)=12 ✓
✅

Interview tip: Always present all four approaches in sequence — naïve → memoised → tabulation → space-optimised. Walk through the dp array by hand for the example input. Interviewers aren't just testing whether you can code it; they're watching whether you can reason about it. The space optimisation — collapsing the dp array into two variables — shows you understand that only the last two states ever matter. That insight also unlocks Climbing Stairs, Fibonacci, and Minimum Cost Climbing Stairs instantly. One pattern, dozens of problems.
