Binary Search

The art of halving your problem at every step — devastatingly fast on sorted data, and impossibly elegant in its simplicity.

Formal Definition

What is Binary Search?

Binary Search is a search algorithm that finds the position of a target value within a sorted array by repeatedly halving the search interval. At each iteration it compares the target with the middle element: if equal, the search is complete; if the target is less, it continues in the left half; if greater, in the right half. This repeats until the element is found or the interval becomes empty, yielding a time complexity of O(log₂ n).

— Standard Computer Science Definition

Key insight: for an array of 1,000,000 elements, binary search finds the answer in at most 20 comparisons. Linear search could need up to 1,000,000.

2
Explanation

The Divide-and-Conquer Idea

Binary search exploits one critical property of sorted arrays: if you look at the middle element and your target is smaller, the target cannot possibly be in the right half. So you discard it entirely and repeat on the left half — and vice versa.

With every comparison you halve the remaining search space. Starting with 1,024 elements: step 1 → 512, step 2 → 256, step 3 → 128 … step 10 → 1. This halving pattern gives O(log₂ n) — the number of times you can halve n before reaching 1.

⚠️
Critical requirement: Binary search only works on a sorted collection. On unsorted data you must sort first (O(n log n)) or fall back to linear search (O(n)).

The algorithm tracks three indices: lo (left boundary), hi (right boundary), and mid = Math.floor((lo + hi) / 2). The loop continues while lo <= hi. If the loop exits without finding the target, it returns -1.

Code Snippet

Binary Search in JavaScript

Both the iterative and recursive approaches, with a step-by-step trace in the comments.

JavaScript — Iterative & Recursive 
// ── Iterative Binary Search — preferred in practice ──
function binarySearch(arr, target) {
  let lo = 0;
  let hi = arr.length - 1;

  while (lo <= hi) {
    const mid = Math.floor((lo + hi) / 2);

    if (arr[mid] === target) {
      return mid;           // ✅ found — return index
    } else if (arr[mid] < target) {
      lo = mid + 1;         // target is in the RIGHT half
    } else {
      hi = mid - 1;         // target is in the LEFT half
    }
  }

  return -1;              // ❌ not found
}

// ── Recursive Binary Search ───────────────────────────
function binarySearchRec(arr, target, lo = 0, hi = arr.length - 1) {
  if (lo > hi) return -1;            // base case: not found

  const mid = Math.floor((lo + hi) / 2);

  if (arr[mid] === target) return mid;
  if (arr[mid] < target)
    return binarySearchRec(arr, target, mid + 1, hi);  // right half
  return   binarySearchRec(arr, target, lo, mid - 1);  // left half
}

// ── Demo ─────────────────────────────────────────────
const sorted = [3, 8, 15, 27, 42, 56, 74, 91];
//               [0] [1]  [2]  [3]  [4]  [5]  [6]  [7]

console.log(binarySearch(sorted, 42)); // 4
console.log(binarySearch(sorted, 99)); // -1

Analogy for a 14-year-old

Think of It Like the Number-Guessing Game

🎯 The Number-Guessing Game

You've played this game: "I'm thinking of a number between 1 and 100. Guess it — I'll say higher or lower."

A bad strategy: start guessing 1, 2, 3, 4... You might need 99 guesses. That's linear search.

A smart strategy: always guess the middle. Start with 50. "Too low!" → guess 75. "Too high!" → guess 62. "Too low!" → guess 68...

Every guess cuts the remaining possibilities in half. You can find any number from 1 to 100 in at most 7 guesses. For 1 to 1,000,000? Just 20 guesses. That's O(log n).

But this only works because the numbers are in order. If your friend said "too high" but the number could be anywhere randomly — your strategy breaks completely. That's why binary search requires a sorted array.

🎯 Key Takeaways

  • Binary search only works on sorted arrays — this is non-negotiable.
  • Each step eliminates half the remaining search space.
  • Time complexity is O(log₂ n) — incredibly fast for large datasets.
  • For 1,000,000 items: binary search needs ≤20 steps; linear needs ≤1,000,000.
  • The iterative version is preferred — it avoids call stack overhead from recursion.
  • Use Math.floor((lo + hi) / 2), not (lo + hi) / 2, to avoid off-by-one errors.
