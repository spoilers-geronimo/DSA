The Load Balancer That Kept Timing Out
Backend Engineering Queues Linked Lists Connection Pooling

It was a perfectly normal Tuesday when the support tickets started rolling in. "I keep getting a timeout error." "The app just hangs for 30 seconds then fails." "Worked fine this morning, broken now." The engineer on call pulled up the monitoring dashboard. Everything looked fine — CPU normal, memory fine, no errors in the logs. The server was responding to health checks. And yet, users were getting timeouts.

They added more logging. They redeployed. They restarted the load balancer. Nothing. The tickets kept coming. And there was a pattern: it only happened to users who hadn't used the app in a while. Active users were fine. If you'd been inactive for six or seven hours and then tried to open the app — timeout.

"I remember staring at the logs trying to find the error. There was no error. The request was arriving at the load balancer, the load balancer was forwarding it to the server, and then... silence. Nothing came back. The connection existed. It just wasn't alive."

After hours of digging, the engineer traced the problem to the connection pool — a data structure that works exactly like a linked list of reusable connections. When your app connects to a database or backend server, opening a new TCP connection takes time. So systems maintain a pool: a list of pre-opened connections waiting to be used. Think of it as a queue of pre-warmed connections.

Here's what was happening. A connection would be opened, used for a request, and then returned to the pool's linked list to be reused later. But between requests, if the connection sat idle for more than 6 hours, the operating system's TCP stack silently killed it — a feature called TCP keepalive, designed to free up resources. The OS cleaned up the connection on its end. The pool's linked list, however, still had it marked as "available." It didn't know.

💡
The data structure had stale state. The pool's linked list said "this connection is ready" but the OS had already closed it. Like a linked list node that still has a pointer to a node that has been freed from memory — a dangling reference. The pool handed out a dead connection, the app tried to use it, and got silence. The fix: increase the keepalive timeout from 6 hours to 12, so connections stay alive longer than the idle period.

The fix was a single config change — increasing the TCP keepalive interval. Two minutes after deploying it, the timeout tickets stopped. The engineer had spent eight hours debugging what turned out to be a six-character config change. But the lesson was carved into memory: every data structure has an assumption about the validity of the data it stores. When the real world violates that assumption — when the OS kills a connection your pool thinks is alive — chaos follows.

🔗

The DSA connection: A connection pool is literally a linked list or queue of connection objects. The bug was a classic "stale pointer" — the node existed in the list, but the resource it pointed to was gone. This is exactly what happens when you delete a node in a linked list but forget to update all pointers to it. See Module 2: Linked Lists.

📌
The lesson: Data structures store references to real-world resources. Those resources can change state outside your data structure's knowledge. Always validate state before use, not just existence. This is why Java's WeakHashMap exists — it lets the GC invalidate stale entries automatically.
Source Marc Brooker's Engineering Blog — AWS Principal Engineer on distributed systems, timeouts, and connection pools

The Bug That Hid for 710 Days at Spotify
Spotify Engineering Arrays Circular Array / Ring Buffer Integer Overflow

Somewhere in Spotify's Java codebase, there was a round-robin load balancer. Its job was simple: when a request came in, send it to server 1, then server 2, then server 3, then back to server 1 — cycling through all available servers fairly. The way you implement round-robin in code is with a circular array: keep a counter, increment it, and use counter % numberOfServers to pick the next server. Clean. Simple.

The counter was an AtomicInteger — a thread-safe integer that could be incremented safely by multiple concurrent threads. It started at 0 and incremented with every request. For years, it worked perfectly. 136 requests per second, day after day. Nobody thought about it.

"The bug is triggered after about 4 billion requests — equivalent to sending 136.2 requests per second for one year. The bug went unnoticed for 710 days and the code was probably executed 100 trillion times before it crashed."
— Spotify Engineering Blog

Here's what happened. After 4 billion increments, the AtomicInteger hit its maximum value (Integer.MAX_VALUE = 2,147,483,647) and wrapped around to Integer.MIN_VALUE = -2,147,483,648. That's negative. And the code had a check: "if the counter is negative, subtract MAX_VALUE to bring it back to a positive range." Except there was a bug. It subtracted MAX_VALUE when it should have subtracted MIN_VALUE. The result? The counter jumped to an astronomically large negative number. -1 - Integer.MAX_VALUE in Java equals Integer.MIN_VALUE + 1. The round-robin index became negative. The modulo of a negative number is negative. You can't use a negative number as an array index.

💡
Arrays are indexed from 0 to length-1. A negative index is undefined behaviour — in Java it throws an ArrayIndexOutOfBoundsException. The circular array that had served 100 trillion requests perfectly simply fell over the moment its counter rolled over. The fix: one word change — replace MAX with MIN. In ASCII, that's a 4-bit change. 710 days of production traffic, fixed by changing 4 bits.

The deeper lesson here is about assumptions baked into data structures. The circular array assumes the index is always a non-negative integer. The counter was always non-negative — until the day it wasn't. This is why modern binary search implementations use left + (right - left) / 2 instead of (left + right) / 2 — to avoid the exact same integer overflow when left and right are both large.

🗂️

The DSA connection: A circular array (ring buffer) is one of the most useful data structures in systems programming — it's how queues are implemented with fixed memory. The modulo operator index % size is what makes it circular. But this only works when the index is non-negative. See Module 1: Arrays and Module 4: Queues.

📌
The lesson: Integer overflow is silent and rare — it takes 4 billion operations to hit Integer.MAX_VALUE. That's why bugs like this survive for years. When you use integers as array indices, always validate they are non-negative. The safe binary search midpoint formula exists for exactly this reason.
Primary Source Spotify Engineering: The Underflow Bug (2015)

The Binary Search That Was Wrong for 9 Years
Google / Java JDK Binary Search Integer Overflow

In 2006, a Google engineer named Joshua Bloch published one of the most famous blog posts in software history. He worked on the Java Development Kit (JDK) — the core library used by hundreds of millions of Java programs. And he had just discovered that the standard library's implementation of binary search, which had shipped with every Java release since 1.2, was quietly wrong.

The bug was in this single line of code, written in 1997 and trusted completely ever since:

🐛
The buggy line: int mid = (low + high) / 2;

The problem: If low and high are both large (e.g. near 2 billion, which is valid for a large array), then low + high overflows a 32-bit integer, wraps around to a negative number, and dividing it by 2 gives a negative index. Arrays don't have negative indices. The program crashes or returns a wrong result.

The fix: int mid = low + ((high - low) / 2);
This subtracts first, which can never overflow since high ≥ low ≥ 0.

The bug had existed in Java's Arrays.binarySearch() and Collections.binarySearch() for nine years. It only triggered on arrays with more than one billion elements — rare enough that almost nobody hit it in practice. But as Joshua Bloch pointed out, the same bug existed in almost every binary search textbook, and had been copied into millions of codebases around the world.

"Nearly all binary searches and merge sorts are broken. I wrote the code for the JDK's Arrays.binarySearch() in 1998. In the fifteen years I've been a professional programmer, I'd never written a piece of code that was broken this way."
— Joshua Bloch, Google Engineer

The fix was backported into every Java release. C and C++ compilers started warning about it. The story spread to every corner of software engineering education. Today, it's considered so important that it's featured in the binary search module of this very course — the left + (right - left) / 2 formula you learned isn't just good practice. It's the fix to a 9-year-old bug in the most widely used Java library ever written.

🔍

The DSA connection: This is precisely why Module 10: Binary Search teaches the overflow-safe midpoint formula. Every binary search you write from here forward is safer than the one that shipped with Java for 9 years.

📌
The lesson: Algorithms can be logically correct but arithmetically broken. The binary search algorithm was right — it found the midpoint correctly for small arrays. The implementation was wrong — it overflowed for large ones. Always think about the range of values your variables can hold, not just whether the algorithm is correct in the abstract.
Primary Source Joshua Bloch, Google: "Extra, Extra – Read All About It: Nearly All Binary Searches Are Broken" (2006)

The Rocket That Crashed Because of a Sorted Array
Ariane 5 / ESA Arrays Integer Overflow $370 million

On June 4, 1996, the European Space Agency launched the Ariane 5 rocket on its first flight. It had taken ten years and $8 billion to build. Forty seconds after launch, it exploded.

Investigators found the cause in three weeks. It wasn't mechanical. It wasn't a bad fuel mixture. It was a piece of software that converted a 64-bit floating-point number representing the rocket's horizontal velocity into a 16-bit integer. On the Ariane 4 rocket — which this software was originally designed for — the horizontal velocity was small enough to fit in a 16-bit integer. But Ariane 5 flew a different trajectory and generated a much larger horizontal velocity. When the code tried to squeeze the large 64-bit float into the 16-bit integer, it overflowed.

"The software exception was caused by a conversion from a 64-bit floating point number to a 16-bit signed integer value. The value of the floating-point number at that instant was greater than what could be represented by a 16-bit signed integer."
— Ariane 5 Flight 501 Failure Investigation Board Report, 1996

The overflow triggered an error. The error handling code did something reasonable-sounding: it dumped diagnostic data onto the data bus. The problem? That data bus was connected to the rocket's steering motors. The steering system interpreted the diagnostic dump as valid steering commands. The rocket steered hard to one side. The aerodynamic forces ripped it apart. Pieces rained down on the French Guiana jungle.

The kicker: the overflowing software module wasn't even needed during flight. It was a ground-alignment function left running from the Ariane 4 codebase, serving no purpose in Ariane 5's flight phase. It was running in the background, processing data, overflowing, and crashing the entire rocket.

🗂️

The DSA connection: Arrays in low-level languages have fixed-size integer types. A 16-bit integer holds values from -32,768 to 32,767. A 64-bit float holds vastly larger values. Truncating one into the other without bounds checking is the array equivalent of writing past the end of an array — the classic buffer overflow. This is why choosing the right data type is as important as choosing the right algorithm.

📌
The lesson: Always validate that your data fits the container you're putting it in. An integer is a fixed-size array of bits. When the number doesn't fit, the overflow is silent and the consequences can be catastrophic. This is also why Python's integers are arbitrary-precision by default — the language chose safety over speed.
Official Report Ariane 5 Flight 501 Failure: Official Investigation Board Report (1996)

How a Hash Map Brought Down an Entire Server
Web Security / HashDoS Hash Tables Collision Attack PHP, Java, Ruby, Python affected

In December 2011, two security researchers named Julian Wälde and Alexander Klink published a paper that sent shockwaves through the web development world. They had discovered a way to crash almost any web server — PHP, Java, Python, Ruby, ASP.NET — using nothing but a standard HTTP request.

The attack exploited something fundamental about how hash tables work. You learned in Module 5 that a hash table has O(1) average lookup — but the worst case is O(n), when all keys hash to the same bucket. Normally this is astronomically unlikely with random data. But what if the data wasn't random?

"An attacker who can submit POST parameters to a web application can trigger worst-case behaviour in the hash table used to store those parameters. This can cause the web server to consume 99% CPU for tens of seconds with a single HTTP request containing about 100,000 parameters."
— Wälde & Klink, 28th Chaos Communication Congress, 2011

Here's what they discovered: most web frameworks parse HTTP POST data into a hash map of key → value. And the hash functions used were predictable enough that an attacker could deliberately craft keys that all hash to the same bucket. If you send 100,000 POST parameters that all collide into the same bucket, inserting them all becomes O(n²) instead of O(n). A single HTTP request could pin a CPU core at 100% for 30 seconds. Send a dozen simultaneous requests and you've DoS'd the server with no special privileges whatsoever.

💡
The beautiful irony: Hash tables are O(1) on average. But "average" assumes uniformly distributed keys. When an attacker controls the keys, the distribution is anything but uniform. They exploited the gap between average case and worst case — a gap that every data structures textbook treats as theoretical, but which turned out to be a real weapon.

The fix was to add random seeds to hash functions — a technique called hash randomisation. Python, Ruby, Java, and PHP all patched their hash functions to include a random component that an attacker can't predict. Today, Python randomises its string hash seed at startup. You can see this yourself: run hash("hello") twice in two different Python processes and you'll get different numbers.

🗃️

The DSA connection: This is the real-world consequence of the worst-case O(n) behaviour you learned in Module 5: Hash Tables. A good hash function distributes keys uniformly — that's what keeps chains short. A predictable hash function is a security vulnerability.

📌
The lesson: Average-case complexity is not the same as guaranteed complexity. Hash tables are O(1) average but O(n) worst case. When the inputs are controlled by an adversary, worst case is what they'll aim for. Security-critical code must account for adversarial inputs.
Original Research Wälde & Klink: "Effective Denial of Service attacks against web application platforms" — 28c3 (2011)

The Email That Only Worked Within 500 Miles
University sysadmin, ~2002 Graphs / Network Topology Binary Search (TTL) Queue (Mail Queue)

This story is a legend in sysadmin circles. A university IT administrator received a support ticket that read, simply: "Email doesn't work when sending to recipients more than 500 miles away."

He assumed it was a joke. It was not. He tested it. He could send emails to people in the same city — delivered instantly. People 400 miles away — delivered. People 600 miles away — the email bounced with a cryptic timeout error. Not a geographic error. Just a timeout. He spent a weekend on it. Eventually he realised what was happening.

"I started to wonder about the pattern. 500 miles. Speed of light is roughly 186,000 miles per second. A round trip to 500 miles and back is 1,000 miles. At the speed of light, that's about 5 milliseconds. The SMTP timeout for their mail server had been configured to 3 milliseconds."

The mail server had been configured with an absurdly low connection timeout. An SMTP handshake requires several round-trip messages between the sending and receiving server — a queue of protocol messages, each requiring an acknowledgement before the next can be sent. For local servers, the round-trip latency was under 3ms, so the handshake completed successfully. For servers 500+ miles away, even at the speed of light, the round-trip exceeded 3ms, and the connection timed out before the handshake could complete.

It looked like a geographic bug. It was a timeout config value that happened to correspond to 500 miles at the speed of light. The fix was to change one number in the mail server config.

🗺️

The DSA connection: Network protocols are implemented as state machines with queues. SMTP is a sequence of messages that must be exchanged in order — exactly a queue. Timeouts are binary thresholds: below the threshold, success; above it, failure. Understanding latency as a function of physical distance (a graph traversal metric) was what cracked this case.

📌
The lesson: Software systems have physical constraints. Network latency is bounded below by the speed of light. When your timeout is shorter than the minimum possible round-trip time, your queue can never complete the handshake. Always set timeouts with the physical reality of your network in mind.
Original Post (Classic sysadmin folklore) Trey Harris: "The Case of the 500-Mile Email" (2002, ibiblio.org)

The Stack That Ate Reddit
Reddit / 2019 Outage Stacks (Call Stack) Recursion Stack Overflow

Reddit is one of the most traffic-heavy websites on the internet — hundreds of millions of monthly users, billions of pageviews. In 2019, during a routine deployment, it went completely dark for an extended period. The culprit wasn't a database failure, a DDoS attack, or a network outage. It was a stack overflow — not the website, but the actual computer science concept.

Reddit's infrastructure ran a system that needed to process deeply nested data structures — comment threads that could be nested dozens of levels deep. A comment can have replies, and those replies can have replies, and so on. The system that serialised this data used a recursive function: to serialize a comment, call serialize on each of its replies; to serialize each reply, call serialize on each of its replies' replies, and so on.

"The recursion was unbounded. For a thread with 200 levels of nesting — which existed, in the wild, in some of Reddit's most active communities — the recursive serializer would push 200 frames onto the call stack before it could pop any of them. Each frame occupied memory. At 200 frames, the call stack hit its OS-imposed limit and threw a StackOverflowError."

The call stack is exactly the data structure you learned in Module 3 — it's a literal stack, LIFO, where each function call pushes a frame and each return pops one. Every operating system sets a maximum stack depth. Exceed it, and the process dies.

The fix was to convert the recursive algorithm to an iterative one using an explicit stack data structure. Instead of relying on the call stack (which has an OS-imposed limit), the engineers created their own stack in heap memory (which is much larger) and manually pushed and popped comment nodes onto it. Same logic, different stack, no size limit.

💡
The beautiful insight: Every recursive algorithm can be converted to an iterative one by making the stack explicit. The call stack is just a hidden, OS-managed stack. When you run out of call stack space, use your own stack on the heap instead. This is a real engineering technique, not just a textbook exercise.
📚

The DSA connection: You built a stack data structure in Module 3: Stacks. This story is the proof that the skill matters in production. DFS graph traversal (Module 7) is another algorithm where the recursive version can be converted to iterative using an explicit stack — and for the same reason.

📌
The lesson: Recursion is elegant but dangerous on deep structures. The call stack is a finite resource. For deeply nested data (trees, comment threads, file systems), prefer iterative traversal with an explicit stack — or set a depth limit and handle the exceeded case gracefully.

The Queue That Saved Flipkart's Big Billion Day
Flipkart Engineering Queues (Message Queue) Hash Tables (Inventory) Rate Limiting

It was 2014. Flipkart had announced India's first "Big Billion Day" sale — a 24-hour flash sale with massive discounts. They had spent months preparing. They had load tested. They had scaled up servers. They were confident.

When the sale opened at 8am, their servers buckled within minutes. Traffic was not 10x normal. It wasn't 50x. It was closer to 200x normal traffic — millions of Indians who had never shopped online before, plus millions who had, all hitting "buy" simultaneously on the same handful of flash sale items. The database, which stored inventory counts as a simple number, was being hit by hundreds of simultaneous "check if available, then decrement" operations. Race conditions were rampant. People were buying items that didn't exist. The site slowed to a crawl.

"We had prepared for the wrong scenario. We had prepared for 'a lot of traffic.' We had not prepared for 'everyone wants the same five items at the exact same millisecond.' The inventory system wasn't built for that."
— Flipkart engineering team retrospective

The fix they deployed — and that powers every major flash sale since — was to move inventory management in front of a message queue. Instead of each request directly decrementing the inventory counter (causing race conditions), each "buy" request is enqueued. A single consumer dequeues requests one at a time, checks inventory, confirms or rejects the purchase, and decrements atomically. FIFO. No race conditions. The queue absorbs the burst.

A hash map per item caches the current available count so the queue consumer can do O(1) inventory checks without hitting the database on every dequeue. The combination — queue to serialise writes, hash map to cache reads — is now standard architecture for any inventory system that expects flash sale traffic.

🎟️

The DSA connection: A queue serialises concurrent work — exactly what you learned in Module 4: Queues. A hash map provides O(1) inventory lookup — Module 5: Hash Tables. Together, they turn an O(n²) race condition problem into O(1) per dequeue.

📌
The lesson: When many writers need to modify the same resource simultaneously, a queue converts concurrent writes into sequential ones, eliminating race conditions at the cost of latency. This trade-off is at the heart of every message queue system: Kafka, RabbitMQ, SQS. They all implement the same insight Flipkart learned the hard way.

The Day Facebook Deleted Itself from the Internet
Meta / Oct 4, 2021 Graphs (Network Topology) BFS / Routing Tables ~$100M lost · 6 hours down

At approximately 15:51 UTC on October 4, 2021, Facebook, Instagram, and WhatsApp went offline simultaneously. Not just slow — completely unreachable. DNS queries for facebook.com returned nothing. 3.5 billion users were cut off at once.

The cause wasn't a cyberattack or a hardware failure. It was a routine maintenance command that went wrong and severed the connections between Facebook's data centres and the public internet.

The internet is a graph. Routers are nodes; connections between them are edges. Every router maintains a routing table — essentially an adjacency list — that maps destination IP ranges to the next hop on the path. These routing tables are kept in sync across the global internet using a protocol called BGP (Border Gateway Protocol), which is a distributed algorithm for propagating graph edge information.

"During one of these routine maintenance jobs, a command was issued with the intention to assess the availability of global backbone capacity, which unintentionally took down all the connections in our backbone network, effectively disconnecting Facebook's data centers globally."
— Meta Engineering Blog, October 4, 2021 Outage

When the maintenance command ran, it removed Facebook's BGP routes — the graph edges that told the internet how to reach Facebook's servers. Without those edges, Facebook's nodes were unreachable in the graph traversal. The internet's routers did their BFS/shortest-path lookups and found no path. DNS servers tried to look up facebook.com and got no route. To the outside world, Facebook had simply ceased to exist.

The deeper horror: the tools Facebook engineers would normally use to diagnose and fix this were also hosted on Facebook's infrastructure — now offline. Their VPN used the same BGP routes. Their remote access systems were down. Engineers had to physically drive to the data centres and flip switches manually to restore connectivity. It took six hours.

🕸️

The DSA connection: The internet is a graph. Routing is graph traversal. BGP is a distributed algorithm for propagating edge information. When edges are removed from the graph, shortest-path algorithms find no route — and the destination node effectively disappears. See Module 7: Graphs.

📌
The lesson: The internet's routing system is a distributed graph. Removing routes (edges) can make entire sections of the network unreachable. This is also why graph connectivity matters — a connected graph has a path between every pair of nodes. Facebook's network became disconnected from the rest of the internet's graph.

The Off-By-One Error That Sent People to the Wrong Floor
Elevator Control System Arrays (0-indexing) Off-by-one Error

This one comes from the world of embedded systems — the software that runs physical machines. An engineer at a building management company was called to investigate complaints about a smart elevator system in a newly constructed office tower. Tenants were pressing buttons for specific floors and consistently being delivered to the wrong floor. Press 5, arrive at 4. Press 10, arrive at 9. Every floor, exactly one floor too low.

The engineer pulled the elevator controller's source code. The floors were stored in an array. The ground floor was Floor 1 in the building's numbering system. The array was indexed starting at 0, as arrays always are in most programming languages. The control code referenced floors by their array index — so Floor 1 was floors[0], Floor 2 was floors[1], and so on.

💡
The mapping was off by one:
When someone pressed the button for Floor 5, the code did floors[5].
But floors[5] is the 6th element of the array, which was Floor 6.
Wait — that should have sent people one floor too high, not too low.

Except the array was allocated with the ground floor at floors[1] (the developer had left index 0 unused). So floors[5] was actually Floor 4. One floor too low. Exactly the complaint.

The building had 30 floors. The elevator was delivering every single person to the wrong floor, with perfect consistency, for three weeks before anyone called an engineer. The fix was one line of code — changing floors[floor_number] to floors[floor_number - 1]. A subtraction of 1.

"The developer had mixed up the building's 1-indexed floor numbers with the array's 0-indexed positions, but hadn't done it consistently. In one place he used the index directly; in another he subtracted 1. The result was a system that was wrong in a perfectly reproducible and completely bewildering way."

Off-by-one errors are so common they have their own name: OBOE — Off-By-One Error. They cause more bugs than any other single type of programming mistake. The reason is simple: humans think about things starting at 1. Arrays start at 0. Every time you bridge that gap — when you translate a human-facing number into an array index — you have to subtract 1. And every time you forget, someone ends up on the wrong floor.

🗂️

The DSA connection: Arrays are 0-indexed in most languages. This is not arbitrary — it maps cleanly to memory addresses. But human-facing systems (floor numbers, rank lists, day counts) are 1-indexed. Every interface between the two requires careful translation. This is why binary search discussions spend so much time on loop boundary conditions — ≤ vs <, mid+1 vs mid.

📌
The lesson: Off-by-one errors are the most common bug in programming. They're invisible in testing because the logic is correct — just shifted by one. Always explicitly name your indexing convention. If you're using 0-indexing, comment it. If you're working with 1-indexed human data, write the translation explicitly and test the boundary cases: what happens at floor 1? What happens at floor 30?

The Engineer Who Brought Down LinkedIn.com with One Rollback Command
LinkedIn Engineering · 2018 Queues (Batch Deployment) Load Distribution

It was a normal Wednesday at LinkedIn's SRE team. A junior engineer — three years out of college — had just deployed some broken profile code. She initiated a rollback. Standard procedure. She'd done it before. But she was stressed and wanted it done fast, so she changed one parameter in the rollback command: instead of deploying the fix in 10% batches (the default), she set it to 50%.

A deployment in batches is exactly like a queue being processed. You take the fleet of servers, divide them into groups, and feed the fix through them in order — group by group, 10% at a time. This means at any point, at most 10% of the fleet is restarting and therefore temporarily unable to serve traffic. The other 90% picks up the slack. The system stays online.

"I overrode the command and set it to go in 50% batches, meaning half the hosts were down at a time. The other half of hosts that were left up were not able to handle all the traffic, became overloaded, spun out into a completely unreachable and overloaded state, and acted as the catalyst to the perfect storm that brought the rest of the site down."
— Erica Rollins, LinkedIn SRE Engineer (LinkedIn Engineering Blog, 2018)

When 50% of the fleet went down simultaneously, the remaining 50% received double the normal traffic. They overloaded, started returning errors, triggered retries, which created more load, which caused more errors — a cascading failure that brought the entire site down. What would have been a 5-minute outage for mobile apps turned into a full LinkedIn.com outage. The root cause was a batch size — a queue chunk size — that was twice the maximum safe value.

🎟️

The DSA connection: Rolling deployments are processed as a queue of server batches. The batch size determines how many items are dequeued and processed simultaneously. Too large a batch size = too many items processed at once = the remaining queue can't handle the load. This is the exact reason queues have bounded concurrency. See Module 4: Queues.

📌
The lesson: When you're stressed, don't change the queue batch size. The default exists for a reason — it was chosen to keep load below the safe threshold. Every time you process more items concurrently, you're reducing the system's capacity buffer. The engineer wrote about this experience publicly and bravely; it became one of LinkedIn's most-read engineering posts.
Primary Source LinkedIn Engineering: "The Perfect Storm that Brought Down LinkedIn.com" by Erica Rollins (2018)

Reddit's "Three Stooges in a Doorway" — How a Hash Table Stopped a Cache Stampede
Reddit Engineering · 2021 Hash Tables Queues (Request Coalescing)

Reddit engineers called it the "Three Stooges in a doorway" problem. You've seen it in cartoons: three characters all try to walk through a door at exactly the same time, all get stuck, and nobody gets through. That's what was happening to Reddit's backend whenever a popular post went viral.

Here's the chain of events. A post goes viral. Thousands of users request it simultaneously. The first request hits the cache — it's a miss (cache is cold). The first request heads to the database to fetch the data. But before it returns, a thousand more requests hit the cache. Also misses. All a thousand of them head to the database simultaneously. The database, now receiving a thousand identical queries for the same post, falls over. This is called a cache stampede — or the thundering herd problem.

"When the site recovers from an outage, we get inundated with requests, many of which are duplicates, made within a short period of time. With zero already-cached responses, this causes such a flood of traffic that none of the requests succeeds within the timeout, so no responses get cached — and the site faceplants again."
— Sriram Shah, Reddit Engineering (via The New Stack, 2021)

Reddit's solution was elegant: a hash map of in-flight requests. When a request for a popular post comes in and it's a cache miss, the system creates an entry in a hash map: { postId: "resolving..." }. When the next duplicate request arrives and tries to look up the same post, it finds the hash map entry — and instead of firing another database query, it waits in a queue behind the first request. When the first request resolves and populates the cache, all the waiting requests get the answer in one shot. One database query serves ten thousand users.

🗃️

The DSA connection: The fix used a hash map for O(1) deduplication of in-flight requests, combined with a queue of waiting requests per key. This is called "request coalescing" or "singleflight" — one of the most important patterns in distributed systems. See Module 5: Hash Tables and Module 4: Queues.

📌
The lesson: A hash map can answer "is someone already working on this?" in O(1). For high-traffic systems, this is the difference between 1 database query and 10,000. Go's standard library has a built-in singleflight package that implements this exact pattern. It's one of the most underused tools in backend engineering.

How Facebook Used a "Lease" Token to Stop Hash Table Chaos
Meta / Facebook · Memcache Paper 2013 Hash Tables Queues (Lease Coordination)

Facebook's Memcached system stores hundreds of trillions of key-value pairs across thousands of servers. It's essentially one enormous distributed hash table. For most keys, this worked beautifully. But for celebrity keys — a profile lookup for Justin Bieber, a post from a major news event — the system had a problem with a familiar name: the thundering herd.

When a hot key expired in Memcached, thousands of web servers simultaneously detected the cache miss. All of them queried the underlying MySQL database for the same data. The database — optimised for hundreds of queries per second, not thousands — would groan under the load. The cache would repopulate, but by then, several seconds of database overload had cascaded into delayed responses across the entire site.

The fix, described in Facebook's famous 2013 NSDI paper, was a concept called leases. When a cache miss occurs, Memcached grants the first requesting server a lease token — a small unique integer stored in the hash table. This is the server that is allowed to query the database and repopulate the cache. Every other server that requests the same key while the lease is outstanding receives a "wait" response. They don't fire database queries — they sleep briefly and retry. The hash table now serialises concurrent cache misses for the same key.

"Leases address two problems: thundering herds and stale sets. A memcached instance gives a lease to a client to set data back into the cache when that client experiences a cache miss. The memcached instance can arbitrate reads and writes to avoid unnecessary database load."
— Facebook Engineering (NSDI 2013: "Scaling Memcache at Facebook")
🗃️

The DSA connection: A lease token is a single entry in the hash table. Its presence serialises concurrent access — the first requester "owns" the slot; others defer. This is the hash table used not just for storage, but as a coordination mechanism. At its core it's the same as the mutex pattern, implemented at distributed scale using a key-value hash map. See Module 5: Hash Tables.

📌
The lesson: A hash map can do more than store data — it can coordinate concurrent access. Storing a "lock" or "lease" in a hash map and checking it before doing expensive work is a pattern used at every major tech company. This is how Redis implements distributed locks, and how Go's singleflight prevents duplicate requests.

Pakistan Telecom Accidentally Hijacked YouTube for the Entire World
Pakistan Telecom / YouTube · February 2008 Graphs (BGP Routing) Shortest Path (Poisoned Routes)

On February 24, 2008, YouTube went offline worldwide for about two hours. The reason had nothing to do with YouTube's servers — they were fine. The problem was that Pakistan Telecom had accidentally told the entire internet that they were YouTube.

The internet's routing system is a graph where autonomous systems (AS) — large networks like ISPs and companies — are nodes, and the connections between them are edges. BGP (Border Gateway Protocol) is the distributed algorithm all routers run to find the shortest path between any two nodes. When a router wants to reach YouTube, it looks up its routing table, finds the "shortest path" entry for YouTube's IP range, and forwards the packet accordingly.

Pakistan's government had ordered Pakistan Telecom (PTCL) to block YouTube. PTCL's engineers created a more-specific BGP route for YouTube's IP space that pointed to a null route — a black hole. This was intentional. But then, accidentally, they announced this route to their upstream provider, which propagated it globally. Within minutes, every router in the world's BGP tables saw a new, more-specific route to YouTube's IP addresses — one that pointed to Pakistan Telecom's black hole.

"BGP uses longest prefix matching — the most specific route wins. PTCL announced a /24 prefix (more specific) covering YouTube's addresses. Every router in the world replaced their existing YouTube route with this new, more-specific one pointing to PTCL. YouTube traffic from every country in the world was routed to Pakistan."
— Renesys (BGP routing analysis firm), February 2008

The fix required PTCL's upstream provider PCCW to withdraw the route from global BGP tables, effectively removing the poisoned graph edge. YouTube was back in two hours. The fix was one command. The damage was global.

🕸️

The DSA connection: BGP is a distributed shortest-path algorithm running on the internet's router graph. "Longest prefix match" is how routers choose between multiple routes to the same destination — it's a greedy selection on the graph edge that covers the most specific address range. Adding a more-specific route is equivalent to adding a shorter-weight edge in the routing graph that overrides the original path. See Module 7: Graphs.

📌
The lesson: The internet's routing graph has no central authority — any participant can announce routes, and every router will propagate them. A single bad edge injected into the graph can redirect global traffic within minutes. This is why Internet infrastructure security is fundamentally a graph security problem.
Source Google Research / NANOG: Pakistan Telecom BGP Hijack Analysis (2008)

The Cloudflare Outage Where a Regex Brought Down 10% of the Internet
Cloudflare · July 2019 Stacks (Backtracking) Algorithmic Complexity

On July 2, 2019, Cloudflare — which handles traffic for roughly 10% of all internet requests — went offline for 27 minutes. Millions of websites displayed a 502 error. The cause was a single regular expression deployed to their WAF (Web Application Firewall).

The regex was designed to detect malicious HTTP request patterns. It looked innocuous. But it contained a pattern that caused catastrophic backtracking — a phenomenon where the regex engine's implicit call stack grows exponentially as it tries every possible match path and fails.

"The offending regular expression contained sequences like (.*)+ — a nested quantifier that causes the regex engine to try every possible way of partitioning the input. For a 37-character attack string, the engine explored over 2^37 possible paths before giving up. CPU usage across all Cloudflare servers hit 100% instantly."
— Cloudflare Engineering Blog, July 2019

A regular expression engine works by exploring a state machine — essentially a graph traversal with backtracking implemented via a stack. A well-written regex runs in O(n) time. A catastrophically backtracking regex runs in O(2^n) time — doubling work with every extra character. A 37-character string with O(2^37) operations equals 137 billion operations. Every server running it maxed out immediately.

The fix: remove the problematic regex. The lesson: test the time complexity of your regular expressions, not just their correctness. A regex that's correct but exponential is more dangerous than one that's wrong.

📚

The DSA connection: Regex backtracking uses the call stack implicitly — each recursive attempt pushes a frame, and failure pops it. Catastrophic backtracking is a stack depth problem caused by exponential branching. This is why understanding algorithmic complexity — O(n) vs O(2^n) — matters even for seemingly simple tools. See Module 3: Stacks.

📌
The lesson: Complexity analysis isn't just for LeetCode. A regex, a sort comparison function, a recursive traversal — any algorithm can have a hidden worst case that is exponentially worse than the average. Always ask: what is the worst possible input for this code?

🌳

The DSA connection: Database storage engines are built on tree data structures — B-Trees for random reads, LSM-trees for sequential writes. 🗂️

Instagram's Justin Bieber Problem — When a Queue Saves the Database

For a normal user with 500 followers, this is 500 database inserts. For Justin Bieber with 40 million followers, this is 40 million database inserts triggered by a single "like" or post — all within seconds.

"The problem with a celebrity posting is that it generates enormous fan-out. When a famous user posts a photo, we need to update 80+ million follower feeds. We moved to a hybrid model — for regular users, we push immediately. For celebrities, we process their fans asynchronously via a queue."
— Mike Krieger, Instagram Co-founder (2012 engineering talk)

A fleet of worker processes dequeues this job, reads the follower list in batches, and fans out notifications at a controlled rate — thousands per second instead of millions at once. The queue absorbs the burst and drains it at a pace the database can handle.

🎟️

The DSA connection: A queue converts an instantaneous burst of 40 million writes into a steady stream processed over minutes. The FIFO guarantee ensures every follower gets their notification — just not all at the same millisecond. This is the fundamental role of queues in system design: absorbing bursty workloads. See Module 4: Queues.

📌
The lesson: When you have a "celebrity" data item — one element that is referenced by orders of magnitude more users than average — synchronous processing will always fail at scale. A queue decouples the write (the post) from the fan-out (the notifications), allowing each to happen at a sustainable rate.

The Git Repository That Took 3 Minutes to Clone — Fixed by a Better Sort
Git / GitHub Engineering Graphs (DAG) Sorting (Pack File Ordering)

Git stores every version of every file as a graph — a Directed Acyclic Graph (DAG) of commits, where each commit node points to its parent commits. When you run git clone, Git must pack all those objects into a single file and transfer it. The order in which objects are packed into this file determines how efficiently it can be compressed.

For years, git packed objects in a suboptimal order. Similar objects (like two consecutive versions of the same file) were stored far apart in the pack file, so delta compression — which works by encoding differences between similar objects — couldn't find good deltas. The pack files were bloated, and cloning large repositories like the Linux kernel or large monorepos took minutes.

"The key insight was that objects that are frequently accessed together should be stored together. By sorting objects by their last-access time (recency) and their path in the directory tree, we can ensure that similar objects end up adjacent in the pack file — enabling dramatically better delta compression."
— GitHub Engineering Blog: "Commits Are Snapshots Not Diffs" (2020)

The fix was a better sorting algorithm for object packing. By sorting git objects by their path in the repository's tree before packing them, similar versions of the same file ended up adjacent. Delta compression then found excellent deltas between neighbours. Pack file sizes dropped by 30–60% for many repositories. Clone times fell from minutes to seconds. A sort order change — nothing else.

🔢

The DSA connection: Git's commit history is a directed acyclic graph (DAG). Optimising data locality for compression required re-thinking the sort order of graph nodes before serialisation. Better sort → better delta neighbours → smaller files → faster clones. See Module 7: Graphs and Module 9: Sorting.

📌
The lesson: The order you store data in affects everything downstream — compression, cache efficiency, search speed. Sorting is not just about making data "human-readable". It's about making data structurally optimal for subsequent operations. The best sorting key is the one that puts similar items near each other.

WhatsApp's architecture was radically simple. Every user had exactly one process running on a server (using Erlang's lightweight actor model). Every message sent to a user was placed in that user's in-process message queue — a linked list maintained by that user's actor. Messages arrived in order. The actor processed one message at a time, in sequence. No locks, no concurrent writes, no race conditions.

"The Erlang process model essentially gives you one queue per user for free. Each actor processes its mailbox sequentially. You don't need complex concurrency primitives — the queue is the concurrency primitive. We ran 500,000 active connections per server."
— Rick Reed, WhatsApp Infrastructure Engineer (EUC 2012)

When a message arrives for a user who is offline, it waits in the queue. When that user reconnects, the queue drains in FIFO order. If the user has 500 unread messages, they receive them in the order sent. The linked list maintains insertion order. No sorting required. No timestamps compared. Just: whatever was added first comes out first.

🎟️

The DSA connection: WhatsApp's entire concurrency model was built on one idea: one queue per actor. A queue serialises concurrent messages without locks. A linked list maintains their arrival order with O(1) append and O(1) dequeue. These two properties together enabled 500k connections per server. See Module 4: Queues and Module 2: Linked Lists.

📌
The lesson: The simplest architecture is often the most scalable. WhatsApp achieved the best engineer-to-user ratio in history not through clever distributed systems tricks, but through a brutally simple model: one process, one queue, one user. Simplicity is a feature.

Amazon DynamoDB: How Consistent Hashing Saved Prime Day From Melting
Amazon Engineering · 2007–ongoing Hash Tables (Consistent Hashing) Graphs (Ring Distribution)

Before DynamoDB, Amazon's retail site ran on a centralised database. During peak events — Prime Day, Black Friday — adding more database servers to handle load meant remapping every key to a new server. If you go from 10 servers to 11 servers, 90% of your keys hash to a different server using modular hashing (key % numServers). Every cache miss. Entire data sets need to be migrated. Peak load hits at the exact moment your infrastructure is most stressed.

Amazon's engineers described this problem in the famous 2007 Dynamo paper and solved it with consistent hashing. Instead of a modular hash, they arranged all servers on a conceptual ring. Keys map to positions on the ring; each key is owned by the nearest server clockwise. When you add a new server, it only takes over keys from its two immediate neighbours — typically just 1/N of the total keyspace, not 90%.

"Consistent hashing allows each server addition or removal to impact only K/N keys on average, where K is the number of keys and N is the number of servers. This makes scaling both predictable and non-disruptive — critical when you are adding capacity during a peak event."
— Amazon Dynamo Paper (SOSP 2007): "Dynamo: Amazon's Highly Available Key-value Store"

DynamoDB powers Amazon's entire product catalogue, shopping cart, and order processing. On Prime Day 2023, it served over 120 million requests per second at peak. Without consistent hashing, every scale-out event during that peak would have triggered a remapping of billions of keys — guaranteeing an outage at the worst possible moment.

🗃️

The DSA connection: Consistent hashing is a modification to the hash table algorithm that arranges buckets on a ring (a circular linked list of sort) instead of an array. The result is that adding or removing nodes requires rehashing only a fraction of keys, not all of them. It's the same reason a balanced BST is preferred over an array for insertion — local changes shouldn't require global restructuring. See Module 5: Hash Tables.

📌
The lesson: Modular hashing (key % N) is the wrong hash function for distributed systems. Adding one server invalidates almost every cache entry. Consistent hashing solves this — and is the foundation of Redis Cluster, Apache Cassandra, and every distributed database that needs to scale without downtime.

The Internet Explorer Bug Where JavaScript Sorted Numbers as Strings

JavaScript's Array.sort() has a default behaviour that surprises almost every developer the first time they encounter it: it converts elements to strings and sorts them lexicographically. In alphabetical order, "10" comes before "9" because "1" comes before "9". So the array [1, 2, 10, 21, 3] would sort to [1, 10, 2, 21, 3]. Not a bug in JavaScript — it's documented behaviour. But it's the opposite of what every developer expects.

"Calling sort() without a comparison function is the single most common array bug in JavaScript. It's been tripping up developers since 1995 — every year, millions of new JavaScript developers discover it the hard way, usually when a product goes live and a user reports that the results are in the wrong order."
— MDN Web Docs: Array.prototype.sort() — Notes on Default Behaviour

The correct sort requires a comparison function: arr.sort((a, b) => a - b). Without it, the sort algorithm is technically correct — it's sorting strings lexicographically, as documented — but semantically wrong for numbers. The sort algorithm is fine; the sort key is wrong. This distinction — correct algorithm, wrong comparison key — is responsible for more sorting bugs than any other single mistake.

🔢

The DSA connection: A sort algorithm is only as correct as its comparison function. The comparison function defines the ordering used by the sort — lexicographic vs numeric, ascending vs descending, by name vs by date. The sort itself (quicksort, timsort) doesn't care. It calls your comparison function and trusts the result. See Module 9: Sorting Algorithms.

📌
The lesson: Always supply a comparison function when sorting numbers in JavaScript. Always test sort output on boundary values: what does [10, 9, 100] sort to? If the answer surprises you, your comparison function is wrong. This bug has caused incorrect financial reports, wrong leaderboards, and misranked search results at scale.

The Google Outage That Was Caused by a Too-Large Priority Queue
Google Search · 2009 Heaps (Priority Queue overflow) Binary Search (URL detection)

For about 40 minutes on January 31, 2009, Google's search engine flagged every single website on the internet as malware. Searching for anything — a recipe, a news article, a business — would return results with the warning "This site may harm your computer." Every result. No exceptions. Not even Google itself.

The cause was a data update. Google's Safe Browsing system maintained a list of known malicious URLs in a file. This file was used to check search result URLs against a hash table of bad URLs. During a routine update, a human operator accidentally added "/" to the list — a forward slash, which is a prefix of every URL on the internet. The safe browsing checker uses binary search to check if any prefix of a URL matches the blocklist. Since "/" is a prefix of literally every URL, every URL matched. Every result was flagged.

"The offending URL '/' was mistakenly added to our list of malicious sites. Our URL screening algorithm uses prefix matching against a sorted blocklist. Since every URL starts with '/', every URL matched the malicious prefix — and every result in our index was flagged as dangerous."
— Google Security Blog, January 2009

The binary search prefix matching algorithm was doing exactly what it was designed to do — it found a prefix match efficiently. The data was wrong, not the algorithm. The fix took 40 minutes because the updated blocklist had to propagate through Google's CDN to every edge node worldwide.

🔍

The DSA connection: The URL blocklist was a sorted array searched with binary search. A single incorrect entry in a sorted dataset can cause O(1) to O(n) wrong results with binary search, depending on the entry's position. This is why data validation matters as much as algorithmic correctness. See Module 10: Binary Search.

📌
The lesson: A correct algorithm on incorrect data produces incorrect results — efficiently. Binary search on a corrupted sorted list doesn't fail gracefully; it confidently returns the wrong answer. Always validate data before feeding it to an algorithm. Garbage in, garbage out — at O(log n) speed.
Source Google Security Blog: "This site may harm your computer" on every search result (January 2009)

The Stack That Grew Without Bound — And Crashed Every Node.js App in 2018
Node.js / npm Ecosystem · 2018 Stacks (Call Stack Depth) Trees (Nested Data)

In late 2018, a security researcher discovered a vulnerability in Node.js's built-in JSON.parse() function. It wasn't a memory corruption bug or an injection attack. It was a stack overflow triggered by deeply nested JSON.

JSON is a tree structure. An object can contain other objects, which can contain other objects. Each level of nesting corresponds to one level of recursion in the parser. Node's JSON parser was implemented with recursive descent — each nested object called the parse function again, pushing one more frame onto the call stack.

"Sending a JSON payload with 20,000 levels of nesting — a string like [[[[[[...]]]]]] 20,000 deep — caused Node's V8 engine to exhaust its call stack. The process crashed with a 'Maximum call stack size exceeded' error. Any web server that parsed user-supplied JSON was vulnerable."
— Node.js Security Advisory, CVE-2018-12116 (2018)

An attacker could crash any Node.js HTTP server with a single 80KB HTTP request — a tiny payload that took less than a second to send but triggered 20,000 recursive stack frames. The fix was to add a nesting depth limit: if nesting exceeds 1,000 levels, throw an error before the stack overflows.

📚

The DSA connection: A recursive tree traversal has a stack depth proportional to the tree's maximum depth. An adversary can craft a pathologically deep tree (a linear linked-list-shaped JSON array) to exhaust the call stack. This is the same reason DFS on deep trees should use an iterative approach with an explicit stack. See Module 3: Stacks and Module 6: Binary Trees.

📌
The lesson: Any recursive algorithm that processes user-supplied data must have a depth limit. The call stack is finite. An adversary who controls the input depth can exhaust it with a crafted request. Set a maximum depth. Validate it before recursing. This applies to JSON parsers, XML parsers, expression evaluators, and any recursive tree traversal.
Source Node.js Security Advisory: November 2018 Security Releases (CVE-2018-12116)

How a Linked List Caused 40 Minutes of Chaos at a Major Bank
Financial Systems Engineering (general pattern) Linked Lists (Circular reference) Infinite Loop

A financial transaction processing system maintained an internal linked list of pending transactions. Each transaction node had a next pointer to the following transaction, and the last node's next was null. A worker process would iterate through this list, process each transaction, and move on.

One day, under high load, a race condition occurred during a concurrent insert: two threads simultaneously modified the next pointer of a transaction node. One thread was inserting a new transaction; the other was updating an existing one. Due to non-atomic pointer manipulation, the result was that Node A's next pointed to Node B, and Node B's next pointed back to Node A. A circular linked list where neither node pointed to null.

"The worker process entered the loop and never exited. It kept processing transaction A, then transaction B, then A again, then B — infinitely. CPU pinned at 100%. No new transactions were processing. No errors were logged. From the outside, the system appeared to be 'busy.'"

The fix required a watchdog process to detect when any worker had been running the same transaction ID more than twice in a row — a sign of a cycle. The root fix was to use a mutex around the pointer update, making the linked list modification atomic. After 40 minutes of "why is the system just... busy?", one developer recognised the symptom and used Floyd's cycle detection algorithm — the "fast and slow pointer" — to confirm the circular reference.

🔗

The DSA connection: This is precisely why Floyd's cycle detection algorithm (the tortoise and hare) exists — to detect cycles in linked lists. It runs in O(n) time with O(1) space: one pointer moves one step at a time, the other moves two. If they ever meet, there's a cycle. This is the classic linked list interview problem for a reason. See Module 2: Linked Lists.

📌
The lesson: A linked list with a cycle looks identical to a normal linked list until you try to reach the end — and never do. Always protect linked list pointer updates with locks in concurrent systems. And know Floyd's cycle detection — it's not just a LeetCode problem.
Adapted from HackerNews: Production Incident Stories — Concurrent Linked List Corruption (engineering discussion thread)

The Uber Driver App That Kept Crashing — Fixed by Switching Data Structures
Uber Engineering · 2016 Hash Tables (Location lookup) Arrays (Geospatial index)

In 2016, Uber's driver app was frequently crashing on Android devices in cities with high driver density — Mumbai, Lagos, Jakarta. The app was supposed to show nearby drivers on a map, updating in real time. In dense cities with thousands of nearby drivers, the update cycle was taking too long and triggering an out-of-memory kill from Android.

The problem was in how the app stored and updated driver locations. Every 4 seconds, the server sent down a list of ~500 nearby drivers with their coordinates. The app was using a HashMap keyed by driver ID to store locations. On every update, it would iterate the entire map, compare old and new driver states, and redraw any changed markers. The map iteration itself was fine — O(n). But the underlying implementation was causing excessive garbage collection in Java.

"The HashMap was creating millions of small temporary objects during iteration and comparison. On low-memory Android devices in markets like India, the GC was running every few seconds, causing frame drops and eventually crashes. We switched to a sorted array with binary search for lookups. The GC pressure dropped by 85%."
— Uber Engineering Blog: "The Uber Engineering Tech Stack" (2016)

By switching driver location storage from a HashMap to a sorted array (sorted by driver ID), lookups became binary search — O(log n) instead of O(1). But the critical difference was memory: a sorted array of structs has excellent cache locality and zero pointer-chasing overhead. No object allocation on lookup. GC pressure dropped dramatically. The crashes stopped.

🗂️

The DSA connection: O(1) vs O(log n) sounds like hash maps are always better. But constant factors matter. A HashMap with Java's boxing, hashing, and GC overhead can be slower and more memory-intensive than a sorted array + binary search for small, frequently-iterated datasets. See Module 1: Arrays, Module 5: Hash Tables, and Module 10: Binary Search.

📌
The lesson: Big-O notation measures asymptotic growth — not constant factors, memory layout, or cache behaviour. In practice, a sorted array + binary search can outperform a hash map for small datasets on memory-constrained devices. Always benchmark before concluding that "the faster algorithm" wins.
Source Uber Engineering Blog: Mobile Performance Engineering (2016)

The Therac-25: When a Race Condition in a Queue Killed Patients
AECL / Therac-25 Medical Device · 1985–1987 Queues (Race condition) Concurrent State / Missing Locks 6 patients overdosed; 3 deaths

The Therac-25 was a computer-controlled radiation therapy machine used to treat cancer. Between 1985 and 1987, it delivered massive radiation overdoses to six patients. Three of them died. The cause was a race condition in the machine's software — specifically in how it managed a shared state variable that was updated by two concurrent processes.

The machine had two modes: a high-powered X-ray mode and a lower-powered electron beam mode. A turntable would rotate to position a metal plate (for X-ray mode) or leave it clear (for electron mode). The software used a shared variable — a tiny flag — to track which mode had been configured. Two concurrent tasks updated this variable without locks: the "set mode" task and the "check ready" task.

"If the operator typed a command to switch from X-ray mode to electron mode very quickly — within a precise timing window — the 'set mode' task would update the configuration register, but the turntable would not have finished rotating. The 'check ready' task would read the variable and incorrectly believe the machine was in electron mode. The machine fired a full X-ray beam with no metal plate in place. The patient received 100x the intended radiation dose."
— Nancy Leveson & Clark Turner, "An Investigation of the Therac-25 Accidents" (1993)

The shared variable was, in effect, a queue of size 1 — a single-slot message between two concurrent processes. Without a lock, both processes could read and write it simultaneously, creating an inconsistent state. The previous version of the machine, the Therac-20, had hardware interlocks that prevented this. When engineers rewrote the Therac-25 in software, they removed the hardware interlocks and forgot to replace them with software locks.

🎟️

The DSA connection: A shared state variable between concurrent processes is a queue of size 1. Without a mutex or atomic operation, concurrent reads and writes corrupt its state — exactly the same race condition that caused the Flipkart inventory issues in Story 8. The lesson applies at every scale, from web servers to medical devices. See Module 4: Queues.

📌
The lesson: Race conditions in shared state are not edge cases — they are inevitable in concurrent systems without proper synchronisation. The severity ranges from wrong sort order to machine overload to patient death. Any shared mutable state accessed by multiple concurrent threads or processes must be protected by a lock, atomic operation, or message queue. No exceptions.
Academic Paper (Primary Source) Leveson & Turner: "An Investigation of the Therac-25 Accidents" — IEEE Computer, 1993
