Formal Definition

Recursion is a programming technique in which a function solves a problem by calling itself on a smaller version of the same problem. A correct recursive function has two essential components: a base case — a condition under which the function returns a result directly without calling itself — and a recursive case — one or more calls to itself with arguments that move closer to the base case. Each recursive call places a new stack frame on the call stack; the stack unwinds as base cases are reached and each frame returns its result to the caller.

Recursion is not a data structure — it is a problem-solving strategy. It is the natural language for problems that have a self-similar structure: trees, graphs, divide-and-conquer algorithms, and any problem where the solution to the whole can be expressed in terms of solutions to smaller instances of the same problem.

Plain English Explanation & Patterns

Every recursive function is built around one core question: "Can I reduce this problem to a smaller version of itself?" If yes, recursion is a natural fit. The structure is always the same: handle the smallest possible input directly (base case), and for everything else, call yourself on the smaller piece and combine the results.

🔄 Iterative Approach

  • Uses explicit loops (for, while)
  • State managed via loop variables
  • Memory: O(1) extra space (no call stack growth)
  • Harder to read for tree/graph problems
  • No risk of stack overflow

🌀 Recursive Approach

  • Function calls itself with a smaller input
  • State managed via call stack frames
  • Memory: O(depth) for call stack
  • Naturally mirrors tree / graph structure
  • Risk of stack overflow on deep inputs

The three laws of recursion — every correct recursive function must follow all three:

  • 1
    It must have a base case. The simplest possible input that can be answered directly, without another recursive call. Without this, the function calls itself forever and crashes the stack.
  • 2
    It must change its state toward the base case. Each recursive call must use a smaller or simpler argument than the one it received. Without this, you never reach the base case.
  • 3
    It must call itself. The recursive call is what makes it recursive. The result of the recursive call is combined with the current frame's local work to produce the answer.
Pattern Description Time Space
Linear recursion One recursive call per frame (factorial, linked list) O(n) O(n)
Binary recursion Two recursive calls per frame (merge sort, binary trees) O(n) O(log n)*
Divide & Conquer Split problem in half, solve each half, combine O(n log n) O(log n)
Exponential recursion Multiple calls with overlapping subproblems (naïve Fibonacci) O(2ⁿ) O(n)
Tail recursion Recursive call is the last operation — can be optimised to O(1) space by a compiler O(n) O(1)*

* Space for tree traversal is O(h) where h = tree height. Tail-call optimisation (TCO) is supported by some languages/compilers.

💡

The call stack is the implicit data structure. Every recursive call pushes a new frame holding the function's local variables and return address. When the base case is reached, frames pop in reverse order, each returning its result to the frame below. This is identical to using an explicit stack — which means every recursive algorithm can be converted to an iterative one by making that stack explicit.

⚠️

Stack overflow: If recursion goes too deep (typically more than ~10,000 frames in JavaScript, ~1,000 in Python), the call stack exceeds its OS-imposed limit and the process crashes. For deeply nested inputs — long linked lists, heavily nested JSON, large trees — prefer an iterative approach with an explicit stack.

Code Snippet

Three progressively more powerful recursive functions — each illustrating a different pattern. Read them as a family: factorial is linear, binary search is divide-and-conquer, and Fibonacci reveals why memoisation exists.

JavaScript — Factorial & Power (Linear Recursion) 
// ─── Factorial: n! = n × (n-1) × (n-2) × … × 1 ─────────────────
// Base case:      n === 0 → return 1
// Recursive case: n × factorial(n - 1)
function factorial(n) {
  if (n === 0) return 1;          // base case
  return n * factorial(n - 1);   // recursive case
}

console.log(factorial(5));   // 120
// Call stack: factorial(5)
//   → factorial(4)
//     → factorial(3)
//       → factorial(2)
//         → factorial(1)
//           → factorial(0) → 1  ← base case
//         ← returns 1×1 = 1
//       ← returns 2×1  = 2
//     ← returns 3×2  = 6
//   ← returns 4×6  = 24
// ← returns 5×24 = 120 ✓

// ─── Power: base^exp using repeated halving ──────────────────────
// Divide & Conquer: x^n = x^(n/2) * x^(n/2)  → O(log n)
function power(base, exp) {
  if (exp === 0) return 1;
  const half = power(base, Math.floor(exp / 2));
  return exp % 2 === 0
    ? half * half             // even exponent
    : half * half * base;     // odd exponent: one extra factor
}

console.log(power(2, 10));   // 1024  — only 4 multiplications, not 10
JavaScript — Fibonacci: Naïve vs Memoised 
// ─── Naïve Fibonacci — O(2ⁿ) — the classic warning ──────────────
// fib(5) calls fib(4) AND fib(3)
// fib(4) calls fib(3) AND fib(2)  ← fib(3) called TWICE
// This exponential branching makes fib(50) take ~2^50 calls.
function fibNaive(n) {
  if (n <= 1) return n;
  return fibNaive(n - 1) + fibNaive(n - 2);
}

// ─── Memoised Fibonacci — O(n) time, O(n) space ──────────────────
// Cache (hash map) stores already-computed results.
// Each subproblem is solved exactly ONCE.
function fib(n, memo = new Map()) {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n);   // cache hit

  const result = fib(n - 1, memo) + fib(n - 2, memo);
  memo.set(n, result);                       // cache result
  return result;
}

console.log(fib(50));   // 12586269025 — instant ✓
// fibNaive(50) would take ~35 seconds. fib(50) takes microseconds.

// ─── Recursive DFS on a tree ─────────────────────────────────────
// The classic recursive pattern: solve left, solve right, combine.
function sumTree(node) {
  if (!node) return 0;                          // base case: null node
  return node.val
    + sumTree(node.left)
    + sumTree(node.right);
}
// Every tree problem reduces to: what do I return for null?
// What do I combine from left and right? That's the template.

Visual Diagram

The call stack builds up as each recursive call is made, then unwinds as each base case returns. This is factorial(4) — four frames pushed, then popped in reverse:

Call Stack — factorial(4) unwinding
← top (currently executing)
factorial(0) → returns 1 (base case)
factorial(1) waiting: 1 × ?
factorial(2) waiting: 2 × ?
factorial(3) waiting: 3 × ?
factorial(4) waiting: 4 × ?

Green = active frame at the top. Each frame waits, frozen, until the frame above it returns. Then it computes its own answer and returns to the frame below.

The recursion tree for naïve fib(5) shows why it's O(2ⁿ) — and why memoisation is the fix. The highlighted nodes are computed twice:

Recursion tree — fib(5)

Red = f(3) computed twice. Blue = f(2) computed three times. Memoisation stores results the first time, eliminating all the duplicate branches instantly.

Plain English Analogy

🪆

Russian Nesting Dolls (Matryoshka)

Picture a set of Russian nesting dolls. You pick up the largest doll and ask: "How many dolls are inside this one?" You can't count them without opening it. So you open it and find a slightly smaller doll inside. You ask the same question again — "How many dolls are inside this one?" You open it. Same question, smaller doll. You keep doing this until you open a doll that has nothing inside — that's your base case. The answer is zero.

Now you start counting on the way back out. The innermost doll contains 0 dolls, so the next one out contains 1 (itself). The next one contains 2. The one after that contains 3. By the time you're back to the largest doll, you've counted them all — without ever having to see the whole picture at once. Each doll only needed to know how many were inside it, and it asked the doll inside to find out.

That is exactly how a recursive function works. Each function call only handles one step, and delegates the rest to the next call — exactly like each doll delegating the inner count to the doll it contains.

  • Each doll = a stack frame
  • "Ask the doll inside" = the recursive call
  • The smallest doll with nothing inside = the base case
  • Counting back out as you close dolls = the call stack unwinding
  • The final count at the outermost doll = the return value of the original call

Real-World Coding Applications

Recursion appears wherever the data has a naturally self-similar or nested structure — anywhere the problem can be broken into smaller versions of itself.

🌳

Tree Traversal (DOM, AST)

Every HTML page is a tree. Browsers parse and render it using recursive descent. Compilers traverse Abstract Syntax Trees recursively — each node calls evaluate on its children and combines the results.

📁

File System Walking

The Unix find command and tools like grep -r recursively walk directory trees. A directory contains files and other directories — a self-similar structure that recursion handles naturally.

🔀

Merge Sort & Quick Sort

Both sorting algorithms are recursive divide-and-conquer. Merge sort splits the array in half, sorts each half recursively, then merges. Quick sort partitions around a pivot and sorts each partition recursively.

🕸️

Graph DFS

Depth-First Search on a graph is naturally recursive: visit a node, then recursively visit each unvisited neighbour. Maze solving, cycle detection, and topological sort all rely on recursive DFS.

🧩

Backtracking (Sudoku, N-Queens)

Backtracking algorithms — solving Sudoku, generating permutations, the N-Queens problem — are recursive: try a choice, recurse deeper, and if it fails, undo it and try the next option. The call stack naturally tracks the undo history.

🗜️

JSON Parsing

JSON is a recursive data format: an object can contain arrays, which can contain objects. Every JSON parser uses recursive descent — the parser for objects calls the parser for values, which calls the parser for objects again when needed.

Classic Interview Question

🔢 Climbing Stairs

You are climbing a staircase with n steps. Each time you can either climb 1 or 2 steps. In how many distinct ways can you climb to the top?

Example: n = 3 → 3 ways: (1+1+1), (1+2), (2+1)

⚡ Easy — LeetCode #70 · the gateway to Dynamic Programming

Why this question? It's the perfect bridge between recursion and dynamic programming. The naïve recursive solution is beautiful and obvious — but O(2ⁿ). The memoised version teaches you to cache subproblems. The bottom-up DP version teaches you to flip the recursion upside down. Three solutions, three paradigms, one problem.

The key insight: To reach step n, you must have come from step n-1 (took 1 step) or step n-2 (took 2 steps). So ways(n) = ways(n-1) + ways(n-2). That's the Fibonacci recurrence in disguise.

  • 1
    Base cases: ways(1) = 1, ways(2) = 2.
  • 2
    Recursive case: ways(n) = ways(n-1) + ways(n-2).
  • 3
    Add a memo map to cache results and drop from O(2ⁿ) to O(n).
  • 4
    Notice only the last two values are ever needed — optimise space to O(1).
JavaScript — Climbing Stairs: Three Approaches 
// ─── Approach 1: Naïve Recursion — O(2ⁿ) ────────────────────────
function climbNaive(n) {
  if (n <= 2) return n;
  return climbNaive(n - 1) + climbNaive(n - 2);
}
// Correct but exponential — times out on n > 40

// ─── Approach 2: Memoised Recursion — O(n) time, O(n) space ─────
function climbMemo(n, memo = {}) {
  if (n <= 2) return n;
  if (memo[n])  return memo[n];    // cache hit
  memo[n] = climbMemo(n - 1, memo) + climbMemo(n - 2, memo);
  return memo[n];
}

// ─── Approach 3: Bottom-up DP — O(n) time, O(1) space ────────────
// Flip the recursion: build from small subproblems up to n.
// We only ever need the last two results — no array needed.
function climbDP(n) {
  if (n <= 2) return n;
  let prev2 = 1, prev1 = 2;

  for (let i = 3; i <= n; i++) {
    const curr = prev1 + prev2;
    prev2 = prev1;
    prev1 = curr;
  }
  return prev1;
}

// ─── Test all three ───────────────────────────────────────────────
console.log(climbNaive(5));   // 8
console.log(climbMemo(5));   // 8
console.log(climbDP(5));     // 8

// n=1→1, n=2→2, n=3→3, n=4→5, n=5→8  (Fibonacci sequence!)
✅

Interview tip: Always present all three approaches in order. Start with the recursive insight — "to reach step n, I came from n-1 or n-2". Show the naïve version, then explain why it's O(2ⁿ) using the recursion tree. Add memoisation and explain it eliminates the duplicate branches. Finally, observe that we only ever look back two steps — flip it bottom-up and eliminate the hash map entirely. This progression from recursion → memoisation → DP is the exact pattern you'll use on dozens of interview problems.
