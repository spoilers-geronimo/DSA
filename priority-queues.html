Formal Definition

A Priority Queue is an abstract data type in which every element is associated with a priority value, and the element with the highest (or lowest) priority is always served first — regardless of insertion order. The canonical implementation is a Binary Heap: a complete binary tree that satisfies the heap property. In a min-heap, every parent node is less than or equal to its children, so the minimum element always sits at the root. In a max-heap, every parent is greater than or equal to its children, so the maximum is always at the root. The entire tree is stored compactly in a plain array — no pointers required.

A binary heap is always a complete binary tree — every level is fully filled except possibly the last, which fills from left to right. This shape guarantee is what makes the array representation possible and keeps all operations tightly bounded at O(log n).

For any node stored at index i in the array:
  •  Left child → (2 × i) + 1
  •  Right child → (2 × i) + 2
  •  Parent → ⌊(i − 1) / 2⌋
These three formulas replace every pointer in the tree — zero extra memory overhead.

Formal Explanation & Operations

🟢 Min-Heap

  • Root always holds the smallest element
  • Every parent ≤ both its children
  • peek() returns the minimum in O(1)
  • Used in: Dijkstra's, A*, merge k sorted lists, task scheduling by earliest deadline
  • Python's heapq module is a min-heap
  • Simulate a max-heap by negating all values

🔴 Max-Heap

  • Root always holds the largest element
  • Every parent ≥ both its children
  • peek() returns the maximum in O(1)
  • Used in: Heap Sort, finding k-largest elements, scheduling by highest priority
  • Java's PriorityQueue with a reversed comparator
  • Identical implementation to min-heap — one comparison flipped
Operation Time Space Description
peek() O(1) O(1) Return the root (min or max) without removing it — it's always at index 0
insert(x) O(log n) O(1) Append to end of array, then sift up: swap with parent while heap property violated
extractMin() / extractMax() O(log n) O(1) Swap root with last element, remove last, then sift down: swap with smaller (or larger) child until restored
heapify(array) O(n) O(1) Build a heap from an unsorted array in-place using Floyd's algorithm — faster than n individual inserts
decreaseKey(i, val) O(log n) O(1) Reduce a node's value and sift up — critical for Dijkstra's algorithm
Search by value O(n) O(1) No ordering beyond the heap property — must scan all elements
Heap Sort (full) O(n log n) O(1) Build max-heap O(n), extract max n times O(n log n) — in-place, no extra memory
💡

Why is insert O(log n)? The new element is placed at the bottom of the tree. A complete binary tree of n nodes has height ⌊log₂ n⌋. In the worst case the element bubbles all the way up to the root — one swap per level — so at most ⌊log₂ n⌋ swaps. Same logic applies to extractMin/Max going down.

⚠️

A heap is NOT fully sorted. The only guarantee is that the root is the min (or max). Siblings at the same level have no ordering relationship with each other — so you cannot do binary search on a heap.

Code Snippet

A complete MinHeap class with insert and extractMin, a MaxHeap showing the single comparison flip, then Heap Sort and a classic interview problem — finding the k largest elements efficiently.

JavaScript — MinHeap (full implementation) 
// ─── MinHeap ──────────────────────────────────────────────────────
// The smallest element is always at index 0 (the root).
// All tree relationships are computed with index arithmetic — no pointers.
class MinHeap {
  constructor() {
    this.heap = [];   // Flat array storing the complete binary tree
  }

  // ── Index helpers ─────────────────────────────────────────────────
  _parent(i) { return Math.floor((i - 1) / 2); }
  _left(i)   { return 2 * i + 1; }
  _right(i)  { return 2 * i + 2; }
  _swap(a, b) {
    [this.heap[a], this.heap[b]] = [this.heap[b], this.heap[a]];
  }

  // ── Core accessors ────────────────────────────────────────────────
  peek()    { return this.heap[0]; }          // O(1) — always the root
  size()    { return this.heap.length; }
  isEmpty() { return this.heap.length === 0; }

  // ── Insert: append at end, then sift UP — O(log n) ───────────────
  insert(val) {
    this.heap.push(val);                        // Step 1: add to bottom
    this._siftUp(this.heap.length - 1);          // Step 2: bubble up
  }

  _siftUp(i) {
    while (i > 0) {
      const p = this._parent(i);
      if (this.heap[p] > this.heap[i]) {
        this._swap(p, i);    // Parent is larger — swap upward
        i = p;               // Move focus to parent position
      } else {
        break;               // Heap property satisfied — stop
      }
    }
  }

  // ── ExtractMin: swap root with last, remove last, sift DOWN — O(log n)
  extractMin() {
    if (this.isEmpty()) throw new Error("Heap is empty");

    const min = this.heap[0];               // Save the minimum (root)
    const last = this.heap.pop();            // Remove last element
    if (!this.isEmpty()) {
      this.heap[0] = last;                   // Put last at root
      this._siftDown(0);                     // Restore heap property
    }
    return min;
  }

  _siftDown(i) {
    const n = this.heap.length;
    while (true) {
      let smallest = i;
      const l = this._left(i);
      const r = this._right(i);

      // Find the smallest among node and its two children
      if (l < n && this.heap[l] < this.heap[smallest]) smallest = l;
      if (r < n && this.heap[r] < this.heap[smallest]) smallest = r;

      if (smallest === i) break;    // Already the smallest — stop
      this._swap(i, smallest);
      i = smallest;                  // Continue sifting down
    }
  }
}

// ─── Usage ────────────────────────────────────────────────────────
const minH = new MinHeap();
[5, 3, 8, 1, 9, 2, 6].forEach(x => minH.insert(x));

console.log(minH.peek());         // 1  ← minimum, always O(1)
console.log(minH.extractMin());   // 1  ← removed from root
console.log(minH.extractMin());   // 2  ← next minimum rises to root
console.log(minH.extractMin());   // 3
console.log(minH.peek());         // 5  ← new minimum after extractions
JavaScript — MaxHeap & Heap Sort 
// ─── MaxHeap ──────────────────────────────────────────────────────
// Identical structure to MinHeap — only the comparisons are flipped.
class MaxHeap {
  constructor() { this.heap = []; }

  _parent(i) { return Math.floor((i - 1) / 2); }
  _left(i)   { return 2 * i + 1; }
  _right(i)  { return 2 * i + 2; }
  _swap(a, b) {
    [this.heap[a], this.heap[b]] = [this.heap[b], this.heap[a]];
  }

  peek()    { return this.heap[0]; }   // O(1) — always the maximum
  size()    { return this.heap.length; }
  isEmpty() { return this.heap.length === 0; }

  insert(val) {
    this.heap.push(val);
    this._siftUp(this.heap.length - 1);
  }

  _siftUp(i) {
    while (i > 0) {
      const p = this._parent(i);
      if (this.heap[p] < this.heap[i]) { // ← flipped: parent smaller → swap
        this._swap(p, i);
        i = p;
      } else break;
    }
  }

  extractMax() {
    if (this.isEmpty()) throw new Error("Heap is empty");
    const max = this.heap[0];
    const last = this.heap.pop();
    if (!this.isEmpty()) { this.heap[0] = last; this._siftDown(0); }
    return max;
  }

  _siftDown(i) {
    const n = this.heap.length;
    while (true) {
      let largest = i;
      const l = this._left(i), r = this._right(i);
      if (l < n && this.heap[l] > this.heap[largest]) largest = l;
      if (r < n && this.heap[r] > this.heap[largest]) largest = r;
      if (largest === i) break;
      this._swap(i, largest);
      i = largest;
    }
  }
}

// ─── Heap Sort — O(n log n) time, O(1) extra space ────────────────
// Phase 1 — Build a max-heap in-place from the raw array: O(n)
// Phase 2 — Repeatedly swap root (max) to sorted end, shrink heap: O(n log n)
function heapSort(arr) {
  const n = arr.length;

  // Build max-heap: start from last non-leaf, sift down each node
  for (let i = Math.floor(n / 2) - 1; i >= 0; i--) {
    siftDown(arr, n, i);
  }

  // Extract elements: largest goes to end, heap shrinks by one each time
  for (let i = n - 1; i > 0; i--) {
    [arr[0], arr[i]] = [arr[i], arr[0]];  // Swap root (max) to sorted end
    siftDown(arr, i, 0);                   // Restore heap on the remainder
  }
  return arr;
}

function siftDown(arr, heapSize, i) {
  let largest = i;
  const l = 2 * i + 1, r = 2 * i + 2;
  if (l < heapSize && arr[l] > arr[largest]) largest = l;
  if (r < heapSize && arr[r] > arr[largest]) largest = r;
  if (largest !== i) {
    [arr[i], arr[largest]] = [arr[largest], arr[i]];
    siftDown(arr, heapSize, largest);
  }
}

console.log(heapSort([5, 3, 8, 1, 9, 2, 6]));
// [1, 2, 3, 5, 6, 8, 9]
JavaScript — K Largest Elements (classic interview problem) 
// ─── Find k-largest elements — O(n log k) ─────────────────────────
// Key insight: keep a MIN-heap of size k.
// The root is always the current k-th largest element seen so far.
// If a new element is larger than the root, it deserves to be in the top k.
// Evict the root, insert the new element. Heap stays size k throughout.
function kLargest(nums, k) {
  const minH = new MinHeap();

  for (const num of nums) {
    if (minH.size() < k) {
      minH.insert(num);                    // Heap not full yet — always insert
    } else if (num > minH.peek()) {
      minH.extractMin();                   // Kick out the smallest of top-k
      minH.insert(num);                    // Replace with new candidate
    }
    // else: num ≤ smallest of top-k → skip it
  }

  // Drain heap — results come out smallest-first, so reverse
  const result = [];
  while (!minH.isEmpty()) result.push(minH.extractMin());
  return result.reverse();   // Largest first
}

console.log(kLargest([3, 1, 5, 12, 2, 11, 7, 9], 3));
// [12, 11, 9]

// Why O(n log k) and not O(n log n)?
// The heap never grows beyond size k, so each insert/extract costs O(log k),
// not O(log n). When k is small (e.g. k=10 from a million items), this is
// dramatically faster than sorting the entire array.

Plain English Analogy

🏥

Min-Heap — The A&E Waiting Room

You walk into A&E with a sprained wrist. Someone else arrives with a broken arm. Then an ambulance rushes in with someone having a heart attack. In a normal queue, you'd be first since you arrived first. But A&E uses triage — whoever is most critically ill is always seen next, no matter when they arrived.

A min-heap works exactly like this. Each patient has a priority number (1 = critical, 10 = minor). The patient with the lowest number (most urgent) is always at the front. The moment a new critical patient arrives and gets a lower priority number, they immediately become the next to be seen — the heap reorganises itself automatically.

  • A new patient being assessed and given a priority number → insert()
  • The doctor calling the most critical patient next → extractMin()
  • Checking who is most critical without calling them yet → peek()
  • A patient's condition suddenly worsening → decreaseKey()
🏆

Max-Heap — A Live Gaming Leaderboard

Picture a live tournament leaderboard that always shows the highest-scoring player at the top. Every time a player scores, their position updates. You don't need to sort everyone from scratch — the leaderboard guarantees the best player is always visible at position 1. When that player is knocked out (extracted), the next best rises to the top automatically.

A max-heap does this in O(log n) per score update. The winner is always at the root in O(1) — no searching, no sorting.

📦

Why Store a Tree Inside an Array?

Imagine a family tree printed on a numbered grid. Person 0 is the grandparent. Their two children are in slots 1 and 2. Those children's children fill slots 3, 4, 5, 6. You never need arrows or labels to figure out who is related to whom — the slot number tells you everything.

A heap uses exactly this trick. The array slot number doubles as the address, and three simple maths formulas — 2i+1, 2i+2, ⌊(i-1)/2⌋ — replace every pointer that a normal tree would need. The result: a tree that fits perfectly inside a flat array with zero wasted memory.

Real-World Coding Applications

Priority queues and heaps are behind some of the most critical algorithms in computer science — from navigation to operating systems to data compression.

🗺️

Dijkstra's Shortest Path

The algorithm behind Google Maps and GPS routing. A min-heap always provides the unvisited node with the smallest known distance in O(log n), making the entire algorithm O((V + E) log V).

⭐

A* Pathfinding

The go-to algorithm in game AI and robotics. Uses a min-heap (open set) to always explore the most promising node next, based on cost + heuristic. Every game with pathfinding uses this.

🖥️

OS Task Scheduler

Operating systems assign each process a priority. A max-heap (or min-heap with inverted priorities) always surfaces the highest-priority runnable process to the CPU with no linear search.

🗜️

Huffman Coding (Data Compression)

ZIP, PNG, and MP3 compression all rely on Huffman encoding. A min-heap repeatedly extracts the two lowest-frequency symbols to merge them — the heap makes this O(n log n) overall.

📊

Live Median / Top-K Streams

Maintaining a running median from a data stream uses two heaps: a max-heap for the lower half and a min-heap for the upper half. The median is always the root of one of them — O(log n) per insert.

🔗

Merge k Sorted Lists

Given k sorted arrays, a min-heap holds one element from each list. Extract the minimum, advance that list's pointer, push the next element. Result is a fully merged sorted output in O(n log k) — used in database query engines and external sorting.
