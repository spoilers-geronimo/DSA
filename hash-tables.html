Formal Definition

A Hash Table (also called a hash map) is a data structure that implements an associative array — a mapping from keys to values. It uses a hash function to compute an integer index (a hash code) from a given key, then stores the associated value at that index in an underlying array (the bucket array). Under a good hash function with a low load factor, insertion, deletion, and lookup all run in O(1) average-case time. When two distinct keys hash to the same index — a collision — the table resolves it via chaining (a linked list per bucket) or open addressing (probing for the next free slot).

The load factor α = n / m (number of entries ÷ number of buckets) governs performance. When α exceeds a threshold (typically 0.7–0.75), the table rehashes: it allocates a larger array and reinserts every entry, keeping average-case O(1) operations intact.

2

Plain English Explanation & Operations

A hash table answers one question blazingly fast: "Does this key exist, and if so, what's its value?" It achieves this by converting any key — a string, a number, an object — into an array index in O(1), then jumping directly to that slot. No searching, no sorting, just arithmetic.

The catch: two different keys can sometimes produce the same index. This is a collision, and every hash table must handle it. The two dominant strategies are:

🔗 Chaining

  • Each bucket holds a linked list of all entries that hash there
  • Insertion is always O(1) — just prepend to the list
  • Lookup scans the bucket's list — O(k) where k = chain length
  • With low load factor, k ≈ 1 → effectively O(1)
  • Used by Java's HashMap, Python's dict

🔎 Open Addressing (Linear Probing)

  • All entries stored in the array itself — no extra nodes
  • On collision, scan forward until an empty slot is found
  • Better cache locality — data stays contiguous
  • Sensitive to clustering at high load factors
  • Used by Python's set, Go's map
Operation Description Average Worst Case
set(key, val) Insert or update a key-value pair O(1) O(n)
get(key) Retrieve value by key O(1) O(n)
delete(key) Remove a key-value pair O(1) O(n)
has(key) Check if a key exists O(1) O(n)
Rehash Resize and reinsert all entries O(n) O(n)

Worst case O(n) occurs when all keys collide into the same bucket — extremely rare with a good hash function.

💡

Why O(1) average? A good hash function distributes keys uniformly. With n entries across m buckets, the expected chain length per bucket is n/m (the load factor). Keeping the load factor below ~0.75 ensures chains stay near length 1, making every lookup essentially a single array access.

⚠️

Hash tables are unordered. Keys have no guaranteed iteration order (though Python 3.7+ preserves insertion order as an implementation detail). If you need sorted keys, use a balanced BST (like a TreeMap in Java) at the cost of O(log n) operations.

Code Snippet

The first snippet shows a hash table built from scratch using chaining, revealing how hashing and collision resolution actually work. The second shows idiomatic usage of JavaScript's built-in Map — the production-grade hash table.

JavaScript — Hash Table from Scratch (Chaining) 
class HashTable {
  constructor(size = 53) {
    this.buckets = new Array(size).fill(null);
    this.size    = size;
  }

  // ─── Hash function: key → index ──────────────────────────────────
  // Uses polynomial rolling hash (prime multiplier reduces collisions)
  _hash(key) {
    let hash = 0;
    for (let i = 0; i < key.length; i++) {
      hash = (hash * 31 + key.charCodeAt(i)) % this.size;
    }
    return hash;
  }

  // O(1) average — insert or update
  set(key, value) {
    const idx = this._hash(key);
    if (!this.buckets[idx]) {
      this.buckets[idx] = [[key, value]];   // start new chain
    } else {
      const existing = this.buckets[idx].find(pair => pair[0] === key);
      if (existing) {
        existing[1] = value;               // update existing key
      } else {
        this.buckets[idx].push([key, value]); // append to chain
      }
    }
  }

  // O(1) average — retrieve value by key
  get(key) {
    const idx   = this._hash(key);
    const chain = this.buckets[idx];
    if (!chain) return undefined;
    const pair  = chain.find(p => p[0] === key);
    return pair ? pair[1] : undefined;
  }

  // O(1) average — check existence
  has(key) { return this.get(key) !== undefined; }
}

// ─── Usage ───────────────────────────────────────────────────────
const ht = new HashTable();
ht.set('name',  'Alice');
ht.set('age',   30);
ht.set('city',  'Bengaluru');
console.log(ht.get('name'));    // 'Alice'
console.log(ht.has('age'));     // true
console.log(ht.get('country')); // undefined
JavaScript — Built-in Map (Production Use) 
// ─── JavaScript's Map: the idiomatic hash table ──────────────────
// Unlike plain objects, Map accepts any type as a key.
const map = new Map();

// set / get / has / delete — all O(1) average
map.set('language', 'JavaScript');
map.set(42,          'the answer');
map.set({ id: 1 },  'object key!');  // any type as key

console.log(map.get('language'));  // 'JavaScript'
console.log(map.has(42));           // true
console.log(map.size);             // 3

map.delete(42);
console.log(map.size);             // 2

// ─── Frequency counter — a classic hash table pattern ───────────
function charFrequency(str) {
  const freq = new Map();
  for (const ch of str) {
    freq.set(ch, (freq.get(ch) ?? 0) + 1);
  }
  return freq;
}

console.log(charFrequency('hello'));
// Map { 'h' => 1, 'e' => 1, 'l' => 2, 'o' => 1 }

// ─── Object literal — simpler but keys must be strings ───────────
const scores = {};
scores['Alice'] = 95;
scores['Bob']   = 87;
console.log('Alice' in scores);   // true  — O(1) key check
console.log(scores['Bob']);        // 87

Plain English Analogy

📚

A Library with a Smart Filing System

Imagine a library with 100 numbered shelves. When a new book arrives, the librarian runs the title through a formula (the hash function) that spits out a shelf number, say shelf 42. The book goes straight to shelf 42. No sorting, no searching — just arithmetic.

When you want to find the book later, you feed the same title through the same formula, get 42, and walk straight to shelf 42. That's O(1) lookup. You didn't scan 100 shelves — you jumped directly to the answer.

But sometimes two books get assigned to the same shelf — a collision. The librarian handles this by keeping a small extra list on the shelf of everything stored there. Since most shelves only have one or two books, finding yours is still nearly instant.

  • Book title = the key
  • Book content = the value
  • The formula (shelf number) = the hash function
  • Numbered shelves = the bucket array
  • Two books on one shelf = a collision
  • Small list on a shared shelf = chaining
  • Expanding to a bigger library = rehashing

Real-World Coding Applications

Hash tables are arguably the single most used data structure in everyday programming.

🐍

Language Dictionaries

Python's dict, JavaScript's Object and Map, Java's HashMap, and Go's map are all hash tables. Variable name lookups in interpreters use hash tables too.

⚡

Database Indexes

Hash indexes in databases (PostgreSQL, MySQL) allow O(1) equality lookups — e.g. WHERE user_id = 42. B-tree indexes are used for range queries; hash indexes win for exact matches.

🌐

DNS Resolution Cache

Your OS caches DNS lookups in a hash table: domain name → IP address. Instead of querying a DNS server every time, it checks the local cache in O(1).

🔐

Password Hashing

Passwords are stored as hashes (bcrypt, Argon2). During login, the entered password is hashed and compared to the stored hash — O(1) comparison, and the original is never stored.

📦

Caching (Redis, Memcached)

In-memory caches like Redis are essentially distributed hash tables. A key (URL, query) maps to a cached response — serving millions of lookups per second in O(1).

🔎

Duplicate Detection

Checking for duplicates in a list, finding unique elements, or counting frequencies all use a hash set or hash map. The pattern is ubiquitous in data pipelines and interview problems alike.

Classic Interview Question

➕ Two Sum

Given an array of integers nums and a target integer target, return the indices of the two numbers that add up to target. You may assume exactly one solution exists and you may not use the same element twice.

Example: nums = [2, 7, 11, 15], target = 9 → [0, 1] (because 2 + 7 = 9)

⚡ Easy — LeetCode #1, the most-asked interview question of all time

Why this question? It's the perfect hash table litmus test. The naïve O(n²) solution is obvious (nested loops). The optimal O(n) solution requires the insight that a hash map can answer "have I seen the complement of this number before?" in O(1). That leap — from brute force to hash-based lookup — is exactly what interviewers want to see.

The approach — complement lookup:

  • 1
    Create an empty hash map: seen maps a value to its index.
  • 2
    Loop through the array. For each element nums[i], compute complement = target - nums[i].
  • 3
    If complement is already in seen, you've found both numbers — return their indices.
  • 4
    Otherwise, store seen[nums[i]] = i and continue.
JavaScript — Two Sum 
// ─── Brute force — O(n²) time, O(1) space ───────────────────────
function twoSumBrute(nums, target) {
  for (let i = 0; i < nums.length; i++) {
    for (let j = i + 1; j < nums.length; j++) {
      if (nums[i] + nums[j] === target) return [i, j];
    }
  }
}

// ─── Optimal — O(n) time, O(n) space ────────────────────────────
function twoSum(nums, target) {
  const seen = new Map();  // value → index

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];

    if (seen.has(complement)) {
      return [seen.get(complement), i];   // found the pair!
    }

    seen.set(nums[i], i);   // store this number for future complements
  }
}

// ─── Walkthrough: nums=[2,7,11,15], target=9 ────────────────────
// i=0: complement = 9-2 = 7.  seen has no 7.  Store {2:0}
// i=1: complement = 9-7 = 2.  seen HAS 2 at index 0!  → return [0,1] ✓

console.log(twoSum([2, 7, 11, 15], 9));   // [0, 1]
console.log(twoSum([3, 2, 4],         6));   // [1, 2]
console.log(twoSum([3, 3],            6));   // [0, 1]
✅

Interview tip: Always start by stating the brute-force O(n²) solution — it shows you understand the problem. Then explain the hash map insight: "Instead of searching the remaining array for the complement, I can pre-store everything I've seen." Mention the space trade-off (O(n) extra space). This two-step structure — brute force first, then optimise — is the ideal interview response pattern for almost any problem.
