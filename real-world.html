The Code Behind the Apps You Use Daily

Real engineers at real companies solved real billion-user scale problems using the exact data structures and algorithms you just learned. Here is the evidence.

🚗
Uber Engineering
Routing Engine · Dispatch System · Surge Pricing
Graphs + Dijkstra's

When Uber engineers built their routing engine in 2014, they explicitly modelled the entire world road network as a weighted directed graph — every intersection is a node, every road segment is an edge weighted by travel time. The Uber engineering blog describes the exact trade-offs they faced:

"One simple example you can try at home is Dijkstra's search algorithm, which has become the foundation for most modern routing algorithms today. However, in a production environment, Dijkstra, or any other algorithm that works on top of an unprocessed graph, is usually too slow… it takes roughly 12 hours to build the contracted graph using all the roads of the world."

The solution? They eventually adopted A* search (Dijkstra with a heuristic) for real-time traffic updates, allowing edge weights to change dynamically as GPS data streams in from millions of active drivers. Their dispatch system also uses bipartite graph matching (the Hungarian Algorithm) to optimally pair riders with drivers — a graph problem at its core.

Surge pricing itself runs a geospatial partitioning system where cities are divided into hexagonal grid cells, each modelled as nodes. Hash maps store demand counts per cell in O(1), and a priority queue (heap) surfaces the highest-demand zones instantly.

Instagram Engineering
Feed Delivery · Photo IDs · Follower Graph
Hash Tables + Sorted IDs

When Instagram scaled from 0 to 14 million users with just 3 engineers, every data structure decision carried enormous weight. Their feed delivery system stored follower lists in Redis linked-list structures — when you post a photo, the Media ID is pushed onto a list for every one of your followers. When a follower opens the app, they fetch a slice of that list and look up the posts in Memcached hash tables in O(1).

Perhaps the most clever algorithmic decision was their custom ID generation scheme. Standard auto-increment IDs would bottleneck at the database level. Instead, Instagram engineers designed a 64-bit ID combining Unix timestamp bits, shard ID bits, and auto-increment bits — making every ID globally unique, time-sortable, and shardable. This is binary arithmetic and hash-based sharding in production.

"Delivery to followers is via lists stored in Redis. The Media ID is pushed on the list for everyone following the user. When it's time to render a feed they take a small number of IDs and look them up in Memcached."

Their social graph (who follows whom) is modelled as a directed graph. The Explore page recommendation system runs BFS-style traversals on this graph to surface content from users two or three degrees away from you — exactly the graph traversal you learned in Module 7.

Linked Lists (Redis) Hash Tables (Memcached) Directed Graphs (Social) Sorting (ID Generation)

X (Twitter) Engineering
For You Feed · Graph Traversal · Inverted Index Search
Graphs + BFS

Twitter's "For You" recommendation pipeline must distil 500 million tweets per day down to roughly 500 personalized recommendations per user. Their engineering blog (open-sourced in 2023) reveals the exact algorithms at work.

"We developed GraphJet, a graph processing engine that maintains a real-time interaction graph between users and Tweets, to execute these traversals. Graph traversals of this type are essential to our Out-of-Network recommendations."

GraphJet maintains a live bipartite graph of user–tweet interactions in memory. When you open the app, the system executes BFS-style random walks on this graph to find tweets that people with similar taste have engaged with — then ranks them using a logistic regression model. The "Real Graph" (their in-network scorer) also predicts engagement likelihood using a weighted graph traversal from your follow network.

Their search system uses a classic inverted index (a hash map from word → list of tweet IDs) to make full-text search across hundreds of billions of tweets work in milliseconds. Building this index required a massively parallelised pipeline where a partitioner (hashing algorithm) divides tweets into shards, and each shard builds a local inverted index concurrently.

Graphs (GraphJet) BFS / Graph Traversal Hash Tables (Inverted Index) Queues (Event Pipelines)

Swiggy & Zomato Engineering
Delivery Route Optimisation · ETA Prediction · Demand Forecasting
Graphs + Shortest Path

Every time you order on Swiggy or Zomato, at least three algorithmic decisions happen before the delivery partner even picks up the food. The delivery routing engine models city roads as a weighted directed graph and runs Dijkstra's or A* to find the fastest path from restaurant to doorstep — accounting for real-time traffic via updated edge weights.

The ETA engine must simultaneously account for restaurant preparation time, traffic, and weather. It maintains a priority queue (min-heap) of pending deliveries sorted by urgency, so the system always dispatches the most time-critical order next. During peak hours (dinner time, cricket matches, monsoon evenings), a demand forecasting pipeline pre-positions delivery partners using heat maps — essentially a spatial hash table counting requests per geohash cell.

Swiggy's engineering blog describes how they cracked the multi-order delivery problem — one delivery partner picking up 2–3 orders simultaneously — as a Vehicle Routing Problem, a generalisation of the Travelling Salesman Problem solved via graph algorithms and dynamic programming heuristics.

Graphs (Road Network) Heaps (Priority Dispatch) Hash Tables (Geospatial) Dynamic Programming (VRP)

Flipkart Engineering
Search Autocomplete · Product Ranking · Flash Sale Traffic
Trees + Hash Tables

Flipkart's search autocomplete — the instant suggestions that appear as you type — is powered by a Trie (prefix tree), a specialised binary-tree variant where each path from root to node spells out a prefix. A min-heap of size K is maintained at each trie node to surface the top-K most popular completions for that prefix in O(K log K). This is exactly the Trie + Heap combination from your Heaps module.

During Flipkart's Big Billion Day sale, traffic spikes 10–50× normal. Their session management uses distributed hash tables (sharded across thousands of servers) to maintain user cart state in O(1), even under extreme concurrency. A naive array-based approach would have been catastrophically slow.

Product ranking for search results uses a merge-sort based pipeline: multiple data sources (inventory, price, ratings, seller score) each produce a sorted list, and a K-way merge using a min-heap combines them into a single ranked feed — the exact Merge K Sorted Lists pattern from Module 8.

Meta (Facebook) Engineering
Friend Suggestions · News Feed · Messenger
Graphs + BFS

"People You May Know" is one of the most impactful features in Meta's history — and it's a textbook graph BFS problem. Starting from your node in the social graph, the system runs a BFS to depth 2, finding all your friends-of-friends. Nodes that appear most frequently in that BFS result (high overlap) are the strongest suggestions. With 3 billion users, this graph has hundreds of billions of edges — requiring custom distributed graph processing systems.

The News Feed ranking system uses a priority queue to maintain the top-scoring posts across all your friends' content streams. Rather than sorting every post ever created by every friend (O(n log n) for potentially millions of posts), Feed maintains a size-K max-heap and streams posts through it — O(n log K) where K is the feed size. This is the Kth Largest Element pattern you solved in Module 8, at 3 billion user scale.

"The whole social graph is traversed using BFS. Friends-of-friends connections are computed at query time using real-time graph traversals across hundreds of billions of edges."

Feature 1 — City Graph & Shortest Path

Model a city as a weighted graph and find the fastest route

Represent city intersections as nodes and roads as weighted edges. Implement Dijkstra's algorithm to find the shortest route from any driver to any rider.

  • Build an adjacency list representation of a 20-node city graph
  • Implement Dijkstra's using a min-heap priority queue
  • Display the shortest path and total travel time
  • Bonus: swap Dijkstra's for A* with Manhattan distance heuristic
Graphs Heaps (Priority Queue) Dijkstra's Algorithm Adjacency List
JavaScript — City Graph + Dijkstra's 
// City represented as adjacency list: node → [{to, weight}]
const city = {
  'A': [{ to: 'B', weight: 4 }, { to: 'C', weight: 2 }],
  'B': [{ to: 'D', weight: 5 }, { to: 'C', weight: 1 }],
  'C': [{ to: 'B', weight: 1 }, { to: 'D', weight: 8 }, { to: 'E', weight: 10 }],
  'D': [{ to: 'E', weight: 2 }],
  'E': []
};

// Dijkstra's — finds shortest path from start to all nodes
function dijkstra(graph, start) {
  const dist    = {};
  const prev    = {};
  const visited = new Set();

  // Initialise all distances to infinity
  for (const node of Object.keys(graph)) {
    dist[node] = Infinity;
    prev[node] = null;
  }
  dist[start] = 0;

  // Min-heap simulation using sorted array (use MinHeap class for production)
  const pq = [[0, start]];  // [distance, node]

  while (pq.length) {
    pq.sort((a, b) => a[0] - b[0]);
    const [d, u] = pq.shift();

    if (visited.has(u)) continue;
    visited.add(u);

    for (const { to, weight } of graph[u] ?? []) {
      const newDist = d + weight;
      if (newDist < dist[to]) {
        dist[to] = newDist;
        prev[to] = u;
        pq.push([newDist, to]);
      }
    }
  }
  return { dist, prev };
}

// Reconstruct path from prev map
function getPath(prev, end) {
  const path = [];
  let curr = end;
  while (curr) { path.unshift(curr); curr = prev[curr]; }
  return path;
}

const { dist, prev } = dijkstra(city, 'A');
console.log('Shortest to E:', dist['E']);   // 9
console.log('Path:', getPath(prev, 'E'));    // ['A','C','B','D','E']

Feature 2 — Driver Matching

Find the nearest available driver to a rider in O(log n)

When a rider requests a ride, MiniRide must find the nearest available driver. A naïve linear scan is O(n) across all drivers — Uber can't afford that at millions of concurrent users. Use a sorted structure for fast nearest-neighbour queries.

  • Store available drivers in a hash map keyed by driver ID
  • Maintain driver distances in a min-heap sorted by proximity to the current ride request
  • Extract the closest driver in O(log n) using heap operations
  • Mark driver as busy and remove from heap — O(log n)
Min-Heap (Closest Driver) Hash Tables (Driver State) Binary Search (Distance Range)
⚡

Feature 3 — Surge Pricing Engine

Detect high-demand zones using a hash map and apply dynamic pricing

Divide the city into grid cells (geohash). Maintain a hash map of cellId → rideRequestCount. When a new request arrives, update the count in O(1). A surge multiplier applies when count exceeds a threshold.

  • Build a frequency counter hash map over the city grid — O(1) update per request
  • Use a max-heap to track the top-3 busiest zones in real time
  • Apply surge multiplier to the price for requests from high-demand cells
  • Decay counts over time using a sliding window (queue-based)
Hash Tables (Demand Count) Max-Heap (Top-K Zones) Queue (Sliding Window)
📋

Feature 4 — Ride Queue & Trip History

A FIFO request queue + searchable sorted trip history

Unmatched ride requests wait in a FIFO queue. Completed trips are stored in a sorted array indexed by timestamp, enabling O(log n) binary search for any historical trip.

  • Use a linked-list backed Queue for O(1) enqueue/dequeue of ride requests
  • Store completed trips sorted by timestamp
  • Use binary search to retrieve trips within a given date range in O(log n)
  • Bonus: a stack of the last 5 trips for the "recent activity" screen
Queues (Ride Requests) Stacks (Recent Activity) Binary Search (Trip History) Sorting (Trip Index)

Section 3 · Roadmap

Where Do You Go From Here?

You have the fundamentals. Here is the exact sequence to go from foundation to job-ready in 4 phases.

Phase 1 — Solidify

⏱ Weeks 1–4
  • Solve 2–3 LeetCode problems per day — all from the topics in this course
  • Implement every data structure from scratch (no built-ins)
  • Build the MiniRide project above
  • Keep a "mistake journal" — write down every bug and why it happened

Phase 2 — Expand

⏱ Weeks 5–10
  • Learn Recursion deeply — it unlocks trees, graphs, DP
  • Start Dynamic Programming with the classic DP patterns (knapsack, LCS, coin change)
  • Learn Two Pointers and Sliding Window patterns
  • Aim for 75–100 LeetCode problems solved total

Phase 3 — System Design

⏱ Weeks 11–16
  • Study System Design basics: CAP theorem, databases, caching
  • Design Twitter, URL shortener, parking lot, rate limiter from scratch
  • Read the engineering blogs listed below — understand why Netflix chose Cassandra
  • Mock system design interviews with peers

Phase 4 — Interview Ready

⏱ Weeks 17–24
  • Grind Blind 75 / NeetCode 150 on LeetCode
  • Do 2 timed mock interviews per week (45 min each)
  • Build one full-stack project showcasing DSA (not CRUD)
  • Apply actively — aim for 5 applications per week
💡

The most important advice: Consistency beats intensity. 45 minutes every day beats 8 hours on Sundays. The compound effect of daily practice is what separates engineers who crack FAANG from those who study for 6 months and give up.
